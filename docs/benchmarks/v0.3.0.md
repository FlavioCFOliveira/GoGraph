# Benchmarks ‚Äî v0.3.0

Headline benchmark and load-test numbers for the GoGraph **v0.3.0**
release. The single-threaded headline figures are consistent with the
[v0.2.0](v0.2.0.md) baseline ‚Äî the one performance change in v0.3.0 was
making **adjacency-list hub operations** (`graph/adjlist` `AddEdge` /
`RemoveAllEdgesFrom`) non-quadratic for high-degree nodes, which is not
part of the headline search set (see
[release-notes/v0.3.0.md](../../release-notes/v0.3.0.md#performance)).

> Re-measure on your own hardware before relying on any absolute
> number; these are medians from one documented run, not a guarantee.

## Headline benchmarks

The canonical regression-gate set (the benchmarks run by
[`scripts/run_headline_bench.sh`](../../scripts/run_headline_bench.sh)
and the PR-time benchstat gate). Each figure is the **median of 6**
runs at `-count=6 -benchmem`.

Command:

```
./scripts/run_headline_bench.sh
```

| Benchmark | Package | ns/op (median) | B/op | allocs/op |
|-----------|---------|---------------:|-----:|----------:|
| `BenchmarkDijkstra_PostWarmup` | `search` | 8,351,750 (‚âà 8.35 ms) | 0 | 0 |
| `BenchmarkDijkstra_Large` | `search` | 8,443,642 (‚âà 8.44 ms) | 1,141,152 (‚âà 1.14 MB) | 4 |
| `BenchmarkBFSDirectionOpt_PowerLaw` | `search` | 30,485,173 (‚âà 30.5 ms) | 0 | 0 |
| `BenchmarkYen_K100` | `search` | 14,569,534 (‚âà 14.57 ms) | 624,812 (‚âà 625 KB) | 3,133 |
| `BenchmarkBrandes_RandomGraph` | `search/centrality` | 8,071,145 (‚âà 8.07 ms) | 96,600 | 1,921 |

The two post-warmup traversal paths (`Dijkstra_PostWarmup`,
`BFSDirectionOpt_PowerLaw`) allocate **zero** bytes per call, confirming
the zero-allocation hot-path mandate still holds for the reusable-state
query forms. All five figures are within run-to-run noise of the
[v0.2.0](v0.2.0.md#headline-benchmarks) medians ‚Äî no headline regression.

## Load test ‚Äî concurrency sweep

CLAUDE.md requires a load-test report "at multiple concurrency levels
(1, 8, 64, 256, 1024 goroutines)". We sweep `GOMAXPROCS` over the
`b.RunParallel` hot-key intern benchmark ‚Äî the tightest shared-state
contention point in the read path, where every goroutine interns the
**same** key and therefore competes for the same cache line. Each
figure is the **median of 5** runs at `-count=5 -benchmem`.

Command:

```
go test -run='^$' -bench='^BenchmarkMapper_Intern_HotKey_Parallel$' \
  -benchmem -cpu=1,8,64,256,1024 -count=5 ./graph/
```

| Goroutines (`GOMAXPROCS`) | ns/op (median) | B/op | allocs/op |
|--------------------------:|---------------:|-----:|----------:|
| 1 | 8.09 | 0 | 0 |
| 8 | 90.19 | 0 | 0 |
| 64 | 89.18 | 0 | 0 |
| 256 | 89.09 | 0 | 0 |
| 1024 | 88.59 | 0 | 0 |

**Reading the sweep.** The single-threaded fast path is ‚âà 8.1 ns/op.
Moving to 8 cores raises per-op latency to ‚âà 90 ns/op because every
goroutine writes to the same hot key's shared state across cores ‚Äî the
expected single-hot-key worst case. The decisive result is the **tail**:
from 8 to 1024 goroutines (a 128√ó over-subscription of the 10 physical
cores) per-op latency stays **flat at ‚âà 88‚Äì90 ns/op with zero
allocations**. The structure degrades gracefully and predictably under
massive over-subscription rather than collapsing ‚Äî the bounded-resource
and lock-contention mandates hold. A realistic workload spreads writes
across distinct keys (sharded), so this single-key figure is a
worst-case floor, not a typical-case cost.
