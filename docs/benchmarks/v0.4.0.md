# Benchmarks — v0.4.0

Headline benchmark and load-test numbers for the GoGraph **v0.4.0**
release.

> Re-measure on your own hardware before relying on any absolute
> number; these are medians from documented runs, not a guarantee.

This release lands targeted **Cypher read-path** performance work
(#1574, #1575) with its own measured deltas, plus a new lock-free
read-only transaction API (#1573). The canonical regression-gate
headline set (`Dijkstra`, `BFS`, `Brandes`, `PageRank`) touches no hot
path changed by this release, so those figures are **carried forward
verbatim from [v0.3.1](v0.3.1.md)** — which remains the authoritative
measured baseline for those paths. The new Cypher figures below were
measured for this release with `benchstat`.

## Cypher read-path improvements (this release)

Measured with `benchstat` at `-count=6`. Negative deltas are
improvements.

### Relationship reconstruction — per-query forward-CSR cache (#1574)

Query: `MATCH (a)-[r]->(b) RETURN count(r)` over a multigraph.

| Graph | sec/op | B/op | allocs/op |
|-------|-------:|-----:|----------:|
| dense | **−88.2 %** | **−95.9 %** | −16.7 % |
| small | −73.6 % | −83.3 % | −17.8 % |

The prior implementation rebuilt the entire forward CSR (`O(V+E)`) on
every result row to read a single slot, making reconstruction
`O(R·(V+E))` per query. A 3-minute DST CPU+heap profile attributed
~60 % of all heap allocations to this path. The snapshot is now built
once per query and reused, with all slot reads served from the cached
snapshot's own slices under the query's single read lock.

### Scalar projection — non-escaping `RowContext` pooling (#1575)

Query: `MATCH (n:N) WHERE n.i>=0 AND n.j>=n.i RETURN n.i,n.j`.

| Metric | Delta |
|--------|------:|
| sec/op | −11.5 % |
| B/op | **−42.1 %** |
| allocs/op | **−26.6 %** |

The per-row context map is recycled through a `sync.Pool` at the two
non-escaping evaluation sites (the `Filter` predicate closure and the
scalar-projection evaluator), gated on the existing analysis flag that
bails on any expression kind that could retain the context past the
call. Only the outer container is recycled; escaped values are never
reused.

## Headline benchmarks (inherited from v0.3.1)

The canonical regression-gate set, unchanged by this release. Each
figure is the **median** of the documented `bench-history.sh` run at
`-benchmem`. The guard-band group (`Dijkstra`, `BFS`, `Brandes`)
remains byte-identical within run-to-run noise — **no regression** at
this version.

| Benchmark | Package | ns/op (median) | B/op | allocs/op |
|-----------|---------|---------------:|-----:|----------:|
| `BenchmarkDijkstra_PostWarmup` | `search` | 8,049,770 (≈ 8.05 ms) | 0 | 0 |
| `BenchmarkDijkstra_Large` | `search` | 8,206,248 (≈ 8.21 ms) | 1,140,413 (≈ 1.14 MB) | 4 |
| `BenchmarkBFSDirectionOpt_PowerLaw` | `search` | 29,376,090 (≈ 29.4 ms) | 0 | 0 |
| `BenchmarkBrandes_RandomGraph` | `search/centrality` | 7,791,073 (≈ 7.79 ms) | 96,600 | 1,921 |
| `BenchmarkPageRank_PowerLaw50K` | `search/centrality` | 7,853,564 (≈ 7.85 ms) | 8,286,761 (≈ 8.29 MB) | 68 |

The two post-warmup traversal paths (`Dijkstra_PostWarmup`,
`BFSDirectionOpt_PowerLaw`) allocate **zero** bytes per call, confirming
the zero-allocation hot-path mandate still holds for the reusable-state
query forms.

## Load test — group-commit concurrent write sweep (inherited from v0.3.1)

The group-commit write path (`wal.Writer.SyncGroup`) is unchanged in
`v0.4.0`. The decisive per-commit latency measurement under concurrency,
on `BenchmarkCommitConcurrent` at `-count=8` (inherited):

| Goroutines | Before (per-commit fsync) | After (group commit) | Delta |
|-----------:|--------------------------:|---------------------:|-------|
| 1 | 3.683 ms | 3.683 ms | flat (p = 0.959) — **no regression** |
| 8 | 3.67 ms | 950 µs | **−74.15 %** |
| 64 | 3.68 ms | 120 µs | **−96.72 %** |
| 256 | 3.67 ms | 31 µs | **−99.16 % (≈ 118× throughput)** |

## Load test — read-path concurrency sweep (inherited from v0.3.1)

CLAUDE.md requires a load-test report "at multiple concurrency levels
(1, 8, 64, 256, 1024 goroutines)". The `b.RunParallel` hot-key intern
benchmark — the tightest shared-state contention point in the read path
— is unchanged in `v0.4.0`. Inherited medians of 5 runs at
`-count=5 -benchmem`:

| Goroutines (`GOMAXPROCS`) | ns/op (median) | B/op | allocs/op |
|--------------------------:|---------------:|-----:|----------:|
| 1 | 8.09 | 0 | 0 |
| 8 | 90.19 | 0 | 0 |
| 64 | 89.18 | 0 | 0 |
| 256 | 89.09 | 0 | 0 |
| 1024 | 88.59 | 0 | 0 |

**Reading the sweep.** From 8 to 1024 goroutines (a 128×
over-subscription of the 10 physical cores) per-op latency stays
**flat at ≈ 88–90 ns/op with zero allocations** — the structure degrades
gracefully under massive over-subscription rather than collapsing, so
the bounded-resource and lock-contention mandates hold.

## Deterministic-simulation evidence

A 5-minute DST swarm (master seed `777`) ran **2 633 seeds with 2 633
passes, 0 failures, 0 panics, 0 data races, and 0 goroutine leaks**, with
all 8 named scenarios covered. The simulator harness itself is exercised
by `BenchmarkSimulatorRun` (`internal/sim`).

## Reproduce

```bash
# Cypher read-path deltas (this release)
go test -run='^$' -bench='^BenchmarkRelProjection$' -benchmem -count=6 ./cypher/

# Headline / guard-band set (inherited baseline)
./scripts/bench-history.sh

# Group-commit concurrent write sweep (median of 8)
go test -run='^$' -bench='^BenchmarkCommitConcurrent$' -benchmem -count=8 ./store/txn/

# Read-path concurrency sweep (median of 5)
go test -run='^$' -bench='^BenchmarkMapper_Intern_HotKey_Parallel$' \
  -benchmem -cpu=1,8,64,256,1024 -count=5 ./graph/

# DST swarm
go run ./cmd/sim -swarm -seed 777
```

Per-change (rather than per-release) tracking lives in
[`history/`](history/) via
[`scripts/bench-history.sh`](../../scripts/bench-history.sh); the
narrative ledger is [`history/LEDGER.md`](history/LEDGER.md). The
authoritative measured baseline for the headline paths above is
[v0.3.1](v0.3.1.md).
