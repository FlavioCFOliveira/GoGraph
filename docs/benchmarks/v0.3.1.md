# Benchmarks — v0.3.1

Headline benchmark and load-test numbers for the GoGraph **v0.3.1**
release. v0.3.1 is a **performance and security-hardening release**: it
delivers measured, benchstat-gated wins on the write, analytics, query,
and read paths while keeping the curated search guard band (Dijkstra /
BFS / Brandes) flat. Every figure below is faithful to a documented run
in [`history/`](history/); the per-change record lives in
[`history/LEDGER.md`](history/LEDGER.md).

> Re-measure on your own hardware before relying on any absolute
> number; these are medians from documented runs, not a guarantee.

## Load test — group-commit concurrent write sweep

The headline reliability change in v0.3.1 is **group commit**: the
per-commit `fsync` is replaced by a leader/follower coalesced fsync
(`wal.Writer.SyncGroup`). The decisive measurement is the per-commit
latency under concurrency, on `BenchmarkCommitConcurrent` at
`-count=8`:

| Goroutines | Before (per-commit fsync) | After (group commit) | Delta |
|-----------:|--------------------------:|---------------------:|-------|
| 1 | 3.683 ms | 3.683 ms | flat (p = 0.959) — **no regression** |
| 8 | 3.67 ms | 950 µs | **−74.15 %** |
| 64 | 3.68 ms | 120 µs | **−96.72 %** |
| 256 | 3.67 ms | 31 µs | **−99.16 % (≈ 118× throughput)** |

**Reading the sweep.** With a per-commit fsync, throughput was capped at
≈ 1/fsync-latency commits per second regardless of core count — every
concurrency level paid the full ≈ 3.67 ms fsync latency. Group commit
coalesces concurrent committers behind a single leader fsync, so
per-commit latency collapses as concurrency rises (the more committers
share a flush window, the less each pays), while the **single-threaded
commit cost is provably flat** — there is no penalty for the
uncontended case. The storage-engine auditor certified the change
preserves all four ACID properties: durability is acknowledged only after
the covering fsync, and a failed group fsync fails every member of the
group.

## Load test — read-path concurrency sweep

CLAUDE.md requires a load-test report "at multiple concurrency levels
(1, 8, 64, 256, 1024 goroutines)". As in [v0.3.0](v0.3.0.md#load-test--concurrency-sweep),
we sweep `GOMAXPROCS` over the `b.RunParallel` hot-key intern benchmark —
the tightest shared-state contention point in the read path, where every
goroutine interns the **same** key and competes for the same cache line.
Each figure is the **median of 5** runs at `-count=5 -benchmem`.

Command:

```
go test -run='^$' -bench='^BenchmarkMapper_Intern_HotKey_Parallel$' \
  -benchmem -cpu=1,8,64,256,1024 -count=5 ./graph/
```

| Goroutines (`GOMAXPROCS`) | ns/op (median) | B/op | allocs/op |
|--------------------------:|---------------:|-----:|----------:|
| 1 | 8.09 | 0 | 0 |
| 8 | 90.19 | 0 | 0 |
| 64 | 89.18 | 0 | 0 |
| 256 | 89.09 | 0 | 0 |
| 1024 | 88.59 | 0 | 0 |

**Reading the sweep.** The single-threaded fast path is ≈ 8.1 ns/op.
Moving to 8 cores raises per-op latency to ≈ 90 ns/op because every
goroutine writes to the same hot key's shared state across cores — the
expected single-hot-key worst case. The decisive result is the **tail**:
from 8 to 1024 goroutines (a 128× over-subscription of the 10 physical
cores) per-op latency stays **flat at ≈ 88–90 ns/op with zero
allocations**. The structure degrades gracefully and predictably under
massive over-subscription rather than collapsing — the bounded-resource
and lock-contention mandates hold. The v0.3.1 lock-free metadata name
registry (#1503) further removes per-id RWMutex traffic on the label and
property-key resolution path; an isolated 8-way read benchmark shows
**−81.57 % time** at contention with identical allocations.

## Reproduce

```
# Headline / guard-band set
./scripts/bench-history.sh

# Group-commit concurrent write sweep (median of 8)
go test -run='^$' -bench='^BenchmarkCommitConcurrent$' \
  -benchmem -count=8 ./store/txn/

# Read-path concurrency sweep (median of 5)
go test -run='^$' -bench='^BenchmarkMapper_Intern_HotKey_Parallel$' \
  -benchmem -cpu=1,8,64,256,1024 -count=5 ./graph/
```
