# Benchmarks ‚ÄĒ v0.6.0

Headline benchmark and load-test numbers for the GoGraph **v0.6.0**
release. v0.6.0 is a **feature release** that widens the analytics and
Cypher surface (four new centrality measures, Cypher map projection,
`shortestPath`/`allShortestPaths` with per-instance relationship typing,
node-label export) and lands a deep reliability and conformance
hardening pass, alongside a broad set of hot-path optimisations whose
per-change record lives in [`history/`](history/) and
[`history/LEDGER.md`](history/LEDGER.md).

The figures below were re-measured for this release with
[`scripts/run_headline_bench.sh`](../../scripts/run_headline_bench.sh)
(`-count=6`, median reported) and the read-path concurrency sweep
(`-count=5`, median reported). They are the authoritative
per-release performance picture for v0.6.0.

> Re-measure on your own hardware before relying on any absolute
> number; these are medians from documented runs, not a guarantee.

The two post-warmup traversal paths (`Dijkstra_PostWarmup`,
`BFSDirectionOpt_PowerLaw`) allocate **zero** bytes per call, confirming
the zero-allocation hot-path mandate still holds for the reusable-state
query forms.

`BenchmarkBrandes_RandomGraph` allocates **62,208 B / 10 allocs per op**.
This reflects the predecessor-arena optimisation (#1515), which had
already landed **at** the `v0.5.0` tag ‚ÄĒ so it is *not* a `v0.5.0 ‚Üí
v0.6.0` delta. The side-by-side measurement in the
[comparison section](#v050--v060-comparison) below confirms Brandes is
**unchanged** between the two releases (identical 62,208 B / 10 allocs in
both). The genuine allocation wins in this cycle are elsewhere ‚ÄĒ most
visibly in `Yen_K100` and in the Cypher `Project` path ‚ÄĒ and are
quantified below.

## Performance impact of this release

This release adds new functionality (centrality measures, map
projection, per-instance relationship typing, node-label export) without
regressing any guard-band path. The hot-path optimisations that landed
since v0.5.0 ‚ÄĒ a columnar edge-property tier, registry copy-on-write,
scan arena reuse, an adaptive-parallelism governor, snapshot write-path
reductions, parallel index backfill, and parallel variants of additional
graph algorithms ‚ÄĒ are tracked per change in [`history/`](history/). The
[`v0.5.0 ‚Üí v0.6.0` comparison](#v050--v060-comparison) below is the
authoritative, **measured** side-by-side evidence for this release.

The four new centrality measures (`Closeness`, `Harmonic`,
`Eigenvector`, `Katz`) are invoked only when a caller requests them and
are not on any traversal, write, or recovery hot path.

## v0.5.0 ‚Üí v0.6.0 comparison

This section is a **real, measured** side-by-side benchmark of the
`v0.5.0` tag against this release's `HEAD`. Both sides were built and run
on the same machine, in the same session, with the **same benchmark
binaries' names**: the `v0.5.0` tree was checked out into a throwaway
`git worktree` (`git worktree add /tmp/gg-v050 v0.5.0`), each benchmark
was run with `go test -bench -benchmem -count=6` on **both** trees, and
the two sample sets were compared with `benchstat` (`old` = `v0.5.0`,
`new` = `HEAD`). The worktree was removed afterwards.

All benchmarks reported here exist **identically in both tags**, so every
comparison is like-for-like. Benchmarks that were renamed, moved, or
added after `v0.5.0` are intentionally excluded ‚ÄĒ there is no honest way
to compare them across the two tags.

### Method

| Property | Value |
|----------|-------|
| Old (base) | `v0.5.0` tag, built in `git worktree` at `/tmp/gg-v050` |
| New | release `HEAD` (231 commits on top of `v0.5.0`) |
| Command | `go test -run='^$' -bench=<set> -benchmem -count=6` |

### Interpretation

- **Genuine wins.** Two large allocation reductions land in this cycle:
  `Yen_K100` drops **‚ąí59% allocs/op** (3,133 ‚Üí 1,280) and **‚ąí27% B/op**,
  from the k-shortest / Yen allocation cleanups (#1668, #1591); the
  Cypher `Project_Throughput` path drops **‚ąí50% allocs/op**
  (20,023 ‚Üí 10,023) and **‚ąí16% B/op** with a **‚ąí25% wall-clock**
  improvement, from partial materialisation for field-extractor
  projections, the reused `Project` input-row header, and lazy node
  materialisation (#1659, #1673, #1588, #1589). The `cypher/exec`
  geomean improves **‚ąí2.94%** in time and **‚ąí7.40%** in allocations.
- **Honest regressions.** `Dijkstra_Large` shows a small, real
  allocation increase (4 ‚Üí 5 allocs/op, +11.8% B/op) and the two
  `Dijkstra` time figures move **+1.4 % / +2.3 %** ‚ÄĒ within a few
  percent and at the edge of run-to-run noise on this machine, but
  reported here as measured rather than hidden. The post-warmup,
  reusable-state traversal paths (`Dijkstra_PostWarmup`,
  `BFSDirectionOpt_PowerLaw`) remain **zero-allocation**, so the
  zero-allocation hot-path mandate is intact. A handful of `cypher/exec`
  throughput micro-benchmarks (`Filter`, `Limit`, `Drain`) move **+1‚Äď2 %**
  in time with **unchanged** allocations; these are noise-level and net
  out against the larger `Project`/`Sort`/`Top` improvements in the
  geomean.
- **Confirmed non-deltas.** `Brandes_RandomGraph` is **identical** in
  both tags (62,208 B / 10 allocs) ‚ÄĒ its arena optimisation predates
  `v0.5.0`. `AllNodesScan_ZeroAlloc` and `RowSlab_Alloc` are unchanged.

### Scope note

The memory-tiering wins of this cycle (columnar edge-property tier,
`propBag`/`labelBag`, frame-of-reference date packing, registry
copy-on-write) are **resident-heap** improvements measured per example
workload ‚ÄĒ most directly in example 26 (per-edge resident memory:
159 ‚Üí 61.8 B/edge) ‚ÄĒ not in the per-op `B/op` of the micro-benchmarks
above, which exercise query and traversal hot paths rather than the
steady-state storage footprint. Those figures are recorded in
[`columnar-edge-properties.md`](columnar-edge-properties.md) and the
memory-audit reports under `docs/`. They are intentionally **not**
forced into the table above, because no single micro-benchmark in both
tags measures them like-for-like.

## Load test ‚ÄĒ read-path concurrency sweep

CLAUDE.md requires a load-test report "at multiple concurrency levels
(1, 8, 64, 256, 1024 goroutines)". As in
[v0.5.0](v0.5.0.md#load-test--read-path-concurrency-sweep), we sweep
`GOMAXPROCS` over the `b.RunParallel` hot-key intern benchmark ‚ÄĒ the
tightest shared-state contention point in the read path, where every
goroutine interns the **same** key and competes for the same cache line.
Each figure is the median of 5 samples.

| Goroutines (`GOMAXPROCS`) | ns/op (median) | B/op | allocs/op |
|--------------------------:|---------------:|-----:|----------:|
| 1 | 10.46 | 0 | 0 |
| 8 | 68.15 | 0 | 0 |
| 64 | 79.86 | 0 | 0 |
| 256 | 86.29 | 0 | 0 |
| 1024 | 87.84 | 0 | 0 |

**Reading the sweep.** From 8 to 1024 goroutines (a 128√ó
over-subscription of the 10 physical cores) per-op latency stays **flat
at ‚Čą 68‚Äď88 ns/op with zero allocations** ‚ÄĒ the structure degrades
gracefully and predictably under massive over-subscription rather than
collapsing, so the bounded-resource and lock-contention mandates hold.
