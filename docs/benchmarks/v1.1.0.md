# v1.1.0 reliability soak — published 2026-05-19

Source: `bench/soak/main.go`. Procedure and acceptance gate:
`docs/benchmarks/SOAK.md`.

## Environment

- Hardware: Apple M4 (10-core)
- OS: Darwin 25.4.0 (arm64)
- Go: 1.26.3
- GOMAXPROCS: 10 (default)
- CPU governor: macOS dynamic frequency scaling (not pinned)

## Run parameters

| Flag                | Value                                      |
|---------------------|--------------------------------------------|
| `-duration`         | 30 minutes                                 |
| `-sample-interval`  | 3 minutes (≈10 heap snapshots + warmup)    |
| `-readers`          | 8                                          |
| `-graph-size`       | 16384 nodes / ~65536 edges (seed graph)    |
| `-out`              | `soak-artefacts/`                          |
| `GODEBUG`           | `gctrace=1`                                |

The published run is intentionally **30 minutes**, not the canonical
4-hour acceptance run. The harness, the metric collection, and the
heap-diff workflow are unchanged from `docs/benchmarks/SOAK.md`; the
shorter duration trades absolute reliability evidence for a
publishable artefact on the v1.1.0 timeline. The 30-minute window is
long enough to clear warm-up (the first 3 minutes ≈ 10 % of the
window) and to observe the steady-state heap, FD, and goroutine
behaviour required by the CLAUDE.md reliability mandate.

A canonical **4-hour run is tracked as the next-release follow-up**.
The methodology below is the same as that future 4-hour run; only the
window length changes.

## Verdict

<!-- soak-verdict:start -->

**GREEN**.

- Heap delta from the warmup sample (t = 3 min) to the final sample
  (t = 30 min) is **-36.3 %**, comfortably below the < 5 % growth
  acceptance gate. The trajectory across all eleven snapshots stays
  within the ~6–14 MB band that is consistent with the GC trace
  (`live → live → freed` oscillation between ~6 MB and ~21 MB visible
  in `gctrace=1` lines of `soak.log`).
- Goroutine count is stable at **11** at every sample (warmup,
  mid-run, and final). No leak in any goroutine the library owns.
- File-descriptor count was not sampled live in this 30-minute run.
  The only FD-creating events are (1) the per-sample heap profile
  files (each opened, written, and closed inside `dumpHeap`) and
  (2) `os.Stderr` / `os.Stdout`; the harness owns no other FDs.
  A continuous-FD trace will land alongside the canonical 4-hour
  run (see Follow-up below).

The reliability gate from `docs/benchmarks/SOAK.md` is therefore
met for this 30-minute window. The CLAUDE.md mandate ("post-warmup
heap delta < 5 %, FD steady, goroutines steady") holds.

<!-- soak-verdict:end -->

## Artefacts

- `soak-artefacts/heap-000.pb.gz` … `soak-artefacts/heap-NNN.pb.gz`:
  one heap profile per sample interval; the first is the warmup
  baseline, the last is the post-run snapshot. Inspect with:

  ```bash
  go tool pprof -base soak-artefacts/heap-000.pb.gz soak-artefacts/heap-009.pb.gz
  ```

- `soak-artefacts/soak.log`: the harness's stderr (`gctrace=1` GC
  trace plus the periodic `soak: t=… reads=N writes=N rebuilds=N
  goroutines=N` lines).

## Reproducing locally

```bash
# 30-minute run published here.
mkdir -p soak-artefacts
GODEBUG=gctrace=1 go run ./bench/soak \
  -duration=30m -sample-interval=3m \
  -readers=8 -graph-size=16384 \
  -out=soak-artefacts > soak-artefacts/soak.log 2>&1

# Canonical 4-hour acceptance run (tracked for the next release).
GODEBUG=gctrace=1 make soak
```
