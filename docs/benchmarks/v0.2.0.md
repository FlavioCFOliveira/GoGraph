# Benchmarks — v0.2.0

Headline benchmark and load-test numbers for the GoGraph **v0.2.0**
release. This is the first per-release benchmark file; the convention
was introduced in v0.2.0, so there is no `v0.1.0.md`. The single-
threaded headline figures are consistent with the v0.1.0 baseline —
the one performance change in v0.2.0 was a memory-footprint improvement
to Personalised PageRank, with bit-for-bit identical results (see
[release-notes/v0.2.0.md](../../release-notes/v0.2.0.md#performance)).

> Re-measure on your own hardware before relying on any absolute
> number; these are medians from one documented run, not a guarantee.

## Headline benchmarks

The canonical regression-gate set (the benchmarks run by
[`scripts/run_headline_bench.sh`](../../scripts/run_headline_bench.sh)
and the PR-time benchstat gate). Each figure is the **median of 6**
runs at `-count=6 -benchmem`.

Command:

```
./scripts/run_headline_bench.sh
```

| Benchmark | Package | ns/op (median) | B/op | allocs/op |
|-----------|---------|---------------:|-----:|----------:|
| `BenchmarkDijkstra_PostWarmup` | `search` | 8,281,922 (≈ 8.28 ms) | 0 | 0 |
| `BenchmarkDijkstra_Large` | `search` | 8,361,657 (≈ 8.36 ms) | 1,140,597 (≈ 1.14 MB) | 4 |
| `BenchmarkBFSDirectionOpt_PowerLaw` | `search` | 30,384,318 (≈ 30.4 ms) | 0 | 0 |
| `BenchmarkYen_K100` | `search` | 14,327,371 (≈ 14.33 ms) | 625,213 (≈ 625 KB) | 3,133 |
| `BenchmarkBrandes_RandomGraph` | `search/centrality` | 7,840,261 (≈ 7.84 ms) | 96,600 | 1,921 |

The two post-warmup traversal paths (`Dijkstra_PostWarmup`,
`BFSDirectionOpt_PowerLaw`) allocate **zero** bytes per call, confirming
the zero-allocation hot-path mandate holds for the reusable-state query
forms.

## Load test — concurrency sweep

CLAUDE.md requires a load-test report "at multiple concurrency levels
(1, 8, 64, 256, 1024 goroutines)". We sweep `GOMAXPROCS` over the
`b.RunParallel` hot-key intern benchmark — the tightest shared-state
contention point in the read path, where every goroutine interns the
**same** key and therefore competes for the same cache line. Each
figure is the **median of 5** runs at `-count=5 -benchmem`.

Command:

```
go test -run='^$' -bench='^BenchmarkMapper_Intern_HotKey_Parallel$' \
  -benchmem -cpu=1,8,64,256,1024 -count=5 ./graph/
```

| Goroutines (`GOMAXPROCS`) | ns/op (median) | B/op | allocs/op |
|--------------------------:|---------------:|-----:|----------:|
| 1 | 7.69 | 0 | 0 |
| 8 | 89.01 | 0 | 0 |
| 64 | 88.93 | 0 | 0 |
| 256 | 88.66 | 0 | 0 |
| 1024 | 88.12 | 0 | 0 |

**Reading the sweep.** The single-threaded fast path is ≈ 7.7 ns/op.
Moving to 8 cores raises per-op latency to ≈ 89 ns/op because every
goroutine writes to the same hot key's shared state across cores — the
expected single-hot-key worst case. The decisive result is the **tail**:
from 8 to 1024 goroutines (a 128× over-subscription of the 10 physical
cores) per-op latency stays **flat at ≈ 88–89 ns/op with zero
allocations**. The structure degrades gracefully and predictably under
massive over-subscription rather than collapsing — the bounded-resource
and lock-contention mandates hold. A realistic workload spreads writes
across distinct keys (sharded), so this single-key figure is a
worst-case floor, not a typical-case cost.

## Reproduce

```
# Headline set (median of 6)
./scripts/run_headline_bench.sh

# Concurrency sweep (median of 5)
go test -run='^$' -bench='^BenchmarkMapper_Intern_HotKey_Parallel$' \
  -benchmem -cpu=1,8,64,256,1024 -count=5 ./graph/
```
