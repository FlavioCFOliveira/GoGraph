# Example 10 — DIMACS 9 routing

## What it demonstrates

Building a deterministic synthetic road-network graph with the DIMACS 9
benchmark harness, freezing it into an immutable CSR snapshot, running a
concrete single-source shortest-paths query (`search.Dijkstra`) that
reconstructs a route, and measuring search performance with a
distribution of random probe queries — throughput and a p50/p95/p99
latency distribution over live heap.

## Domain / scenario

A synthetic road network generated by `dimacs9.Synthetic`. Every vertex
points to its next-numbered neighbours — a chain that keeps the whole
graph reachable from node 0 — plus a few longer "shortcut" edges, with
`int64` weights derived deterministically from the endpoint indices. The
two scale knobs are the vertex and edge counts; the average out-degree is
`edges / vertices` (clamped to a minimum of 2).

**The topology is seed-independent.** The DIMACS 9 synthetic generator
takes *no* seed: destinations and weights are pure arithmetic on the
endpoint indices, so the graph — and therefore every shortest path over
it — is identical on every run and every machine for a fixed
(`-vertices`, `-edges`). The `-seed` flag controls *only* the random
probe workload: which source→target pairs the timed Dijkstra queries run
between, to build the latency distribution. Changing `-seed` changes the
volatile timing telemetry but never the deterministic facts (the route,
its distance, the counts).

## How to run

```sh
go run ./examples/10_dimacs9_routing                                          # small deterministic default
go run ./examples/10_dimacs9_routing -vertices 200000 -edges 1000000 -probes 5000 -seed 7  # observable-scale run
```

## Scale and flags

| Flag | Meaning | Default | Large |
|---|---|---|---|
| `-vertices` | number of road-network vertices (fixes the topology) | `2000` | `200000` |
| `-edges` | number of directed edges (fixes the topology) | `12000` | `1000000` |
| `-target` | destination of the fixed concrete route from node 0 | `50` | `199999` |
| `-probes` | number of random source→target probe queries to time | `200` | `5000` |
| `-seed` | RNG seed for the probe workload **only** (topology is seed-independent) | `1` | `7` |

The default builds a 2,000-node, 12,000-edge network (average out-degree
6) and times 200 probe queries — milliseconds end to end, well under the
60 s short-test budget, yet large enough for the latency distribution to
be meaningful.

## Expected output

The bare `key=value` lines are deterministic *facts*: they are
reproducible for a fixed (`-vertices`, `-edges`) on every run, every
machine, and every `-seed`, and are pinned by the regression test. The
`# `-prefixed lines are volatile *telemetry* (throughput, latency
percentiles, heap): they vary per run, per machine, and — for the latency
distribution — per `-seed`, and are never pinned.

```
config.vertices=2000
config.edges=12000
config.target=50
config.probes=200
config.seed=1
graph.nodes=2000
graph.edges=12000
route.src=0
route.dst=50
route.distance=171
route.hops=8
route.path=0 -> 1 -> 2 -> 17 -> 32 -> 47 -> 48 -> 49 -> 50
reach.from_src=2000
probe.count=200
probe.feasible=200
# build.elapsed=3.577ms
# probe.throughput=6900 queries/s
# probe.latency.p50=111.792µs
# probe.latency.p95=268.875µs
# probe.latency.p99=279.375µs
# mem.heap_alloc=454.92 KiB
```

(The `# ` lines above are illustrative; expect different numbers on your
machine. The fact lines are stable.)

## Evidence it collects

This is a search/path-finding example, so it reports the evidence the
standard prescribes for that subject:

- **Wall-clock latency distribution** over the probe queries: `p50`,
  `p95`, and `p99`, plus the mean **throughput** in queries/s.
- **Live heap** (`mem.heap_alloc`, forced-GC) and the heap growth and
  allocation totals across the build-and-query phase.
- **Build throughput** in nodes/s and edges/s for the CSR snapshot.

The deterministic facts (the reconstructed route, its distance, and the
reachable/feasible counts) prove correctness; the telemetry shows
performance. Scale `-vertices`/`-edges` up and watch the per-query
latency grow with the graph while the route facts stay fixed; raise
`-probes` for a tighter distribution; change `-seed` to confirm the facts
do not move while the timing does.

## Key APIs

- `bench/dimacs9.Synthetic` — generate the deterministic, seed-independent synthetic road-network adjacency list from `(vertices, edges)`.
- `graph/csr.BuildFromAdjList` — freeze the builder into an immutable CSR snapshot; `CSR.Order` / `CSR.Size` report the node and edge counts.
- `graph/adjlist.AdjList.Mapper` — translate between node values and compact `NodeID`s (`Lookup`, `Resolve`).
- `search.Dijkstra` — single-source shortest paths over non-negative weights.
- `search.Distances.Distance` / `search.Distances.Path` — read back the cost and reconstruct the route.
