# Release Notes — GoGraph v0.3.2

## Overview

GoGraph v0.3.2 is the fifth published release of GoGraph, a Go module
for graph **persistence**, **manipulation**, and — above all — **fast
search**, designed to scale from in-memory graphs to graphs that exceed
RAM.

This is a pre-1.0 **PATCH** release with a single, focused purpose: it
**fixes a data-compatibility recovery panic** that could crash the
process when opening an existing on-disk store. It is **API-additive**
over `v0.3.1`: no exported identifier was added, removed, or renamed,
there is **no breaking change**, and there is **no new user-facing
public API**. The change is correctness-only and touches no hot path —
which is precisely why it qualifies as a patch rather than a minor
release.

It is a **strongly recommended upgrade** for anyone running `v0.3.0` or
`v0.3.1`, in particular anyone whose store was first written by `v0.2.0`
or earlier (see [Upgrade notes](#upgrade-notes)).

Under [Semantic Versioning](https://semver.org/), a `0.y.z` version
signals that the public Go API is **not yet stable**: it may change
without a major-version bump while the module matures toward `1.0.0`.
Pin the exact version you depend on.

Both compliance guarantees that define the project continue to hold
without regression at this version:

- **100 % openCypher TCK-compliant at the execution level** — every
  scenario in `cypher/tck/features/` passes, with no `failed`,
  `undefined`, or `pending` step. The execution gate stands at
  **3 897 / 3 897 scenarios** (error-fidelity gate 121). The regression
  gate (`tckExecutionBaseline = 3897` in `cypher/tck/runner_test.go`)
  rejects any change that lowers the count.
- **100 % ACID-compliant** — Atomicity, Consistency, Isolation, and
  Durability hold across the in-memory engine and every persistence
  backend; durability is verified by the WAL recovery tests in
  `store/wal` and `store/recovery` and by the deterministic
  crash-injection battery in `internal/crashinject`.

Install with:

```bash
go get github.com/FlavioCFOliveira/GoGraph@v0.3.2
```

The Go module path is `github.com/FlavioCFOliveira/GoGraph` with **no**
`/vN` suffix, which is Semantic-Import-Versioning-correct for a `0.x`
line. The Go toolchain remains **go1.26.4** (unchanged from `v0.3.1`).

For the full, per-change log see [CHANGELOG.md](../CHANGELOG.md).

## Highlights

- **Fixes a recovery-time crash (data compatibility).** Opening an
  on-disk store written by `v0.2.0` or earlier — or any store whose edge
  layout produced a handle-less prefix on a node — no longer panics with
  `makeslice: cap out of range`. Recovery succeeds under both read and
  write paths.
- **Recommended upgrade for all `v0.3.0` / `v0.3.1` users.** The fix
  restores the ability to open affected stores; no data migration or
  query change is required.
- **Correctness-only patch.** No hot path is touched; performance figures
  are inherited unchanged from `v0.3.1`.

## What changed (behavioural)

This release introduces **no breaking API change and no behavioural
change** to any documented API contract. It changes exactly one
internal code path so that store recovery no longer crashes.

### Recovery panic on the edge fast-path handle column (`graph/adjlist`)

`graph/adjlist.upsertEdgeLocked` could panic with `makeslice: cap out of
range` on the **spare-capacity fast path** when growing the per-edge
handle column for a node that had accrued a **handle-less prefix**.

A node's edges are stored as two length-aligned columns: the neighbour
list and a parallel per-edge handle column, where a `0` slot means "no
handle". When a sequence of handle-less inserts (`AddEdge`) precedes a
handle-bearing insert (`AddEdgeHIfAbsent`), the handle column is left
nil or short (for example length `0`) while the neighbour backing array
grows and keeps spare capacity. The next handle-bearing append then took
the fast path and reallocated the handle column sizing it from
`len(current.handles)` instead of from the neighbour count `oldLen`. For
a column of length `0` growing to length `6`, the computed capacity
`growCap(0) = 4` is **below** the required length, so
`make([]uint64, newLen, <newLen)` panicked.

The fix sizes the new column from `oldLen`, matching the slow path in
the same function:

- `make([]uint64, newLen, growCap(oldLen))` — `growCap(oldLen)` is always
  `>= newLen`;
- the existing copy and zero-fill keep the handle column length-aligned
  with the neighbour list, so leading handle-less slots remain the `0`
  "no handle" sentinel.

#### Why it was a data-compatibility break

The defect was introduced by the amortised-O(1) `AddEdge` hub rewrite
(commit `877e455`) and shipped in `v0.3.0` and `v0.3.1`. It is **absent
in `v0.2.0`**.

`store/snapshot.ApplyCSRToGraph` replays each node's edges during
recovery as a **mix** of handle-less (`AddEdge`) and handle-bearing
(`AddEdgeHIfAbsent`) inserts. A node whose replay produced a handle-less
prefix followed by a handle-bearing edge therefore hit the faulty fast
path **on open**, crashing the process under both read and write
recovery. The practical consequence: a store first written by `v0.2.0`
or earlier could not be reopened by `v0.3.0` / `v0.3.1`. Upgrading to
`v0.3.2` restores the ability to open these stores.

## Performance

This is a **correctness-only PATCH** that does not alter any hot path,
so no fresh benchmarks were run for this release: the working changes
are confined to a single allocation-sizing correction on the edge
fast-path. All headline benchmark and load-test numbers are therefore

[docs/benchmarks/v0.3.2.md](../docs/benchmarks/v0.3.2.md) carries those

on your own hardware before relying on any absolute number.

## Upgrade notes

This release is **API-additive** over `v0.3.1`: no exported identifier
was added, removed, or renamed, there is **no breaking API change**, and
there is **no behavioural change** to any documented API contract.
Upgrading is a drop-in `go get`:

```bash
go get github.com/FlavioCFOliveira/GoGraph@v0.3.2
```

**Who should upgrade:** everyone on `v0.3.0` or `v0.3.1`. The upgrade is
**strongly recommended** if your store was first written by `v0.2.0` or
earlier, because such stores could fail to reopen on `v0.3.0` / `v0.3.1`
with a `makeslice: cap out of range` panic. After upgrading, the
affected stores reopen normally; **no data migration is required** and no
query needs to change.

Because the public API is pre-1.0, continue to **pin the exact version**
you depend on.
