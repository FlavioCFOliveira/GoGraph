## Overview

GoGraph v3.0.1 is a **durability and correctness** patch. Its headline is a
fix for **non-durable node deletion**: before this release a deleted node
could resurrect as an undeletable, label-stripped "ghost" after a store
reopen, because the in-memory tombstone set was never persisted into the
snapshot nor reconstructed on WAL replay. The release also lands an
edge-deletion-hygiene fix, a recovery-ordering correctness fix, a MERGE
cross-type-equality conformance fix, an `int64`-overflow fix in `range()`,
and a build-tooling repair.

The two compliance guarantees that define the project were held on every
commit:

- **100 % openCypher TCK-compliant at the execution level** â€” the gate stayed
  at **3 897/3 897 scenarios** (16 006/16 006 steps) with no `failed`,
  `undefined`, or `pending` steps. The regression gate
  (`tckExecutionBaseline = 3897` in `cypher/tck/runner_test.go`) rejects any
  pull request that lowers the count.
- **100 % ACID-compliant** â€” Atomicity, Consistency, Isolation, and Durability
  hold across the in-memory engine and every persistence backend. This release
  *strengthens* Durability: a committed node or edge deletion now survives a
  process restart on every recovery path (snapshot-only, WAL-only, and
  snapshot + WAL-tail).

There are **no breaking changes** and **no on-disk format break**: existing
WAL, snapshot, and CSR files load unchanged, and a snapshot of a graph that
never deleted a node is byte-identical to one produced by v3.0.0. Upgrade with
`go get github.com/FlavioCFOliveira/GoGraph@v3.0.1`.

For the full, per-change log see [CHANGELOG.md](../CHANGELOG.md).

## Fixed

### Node deletion is now durable across a store reopen

Node deletion is a **tombstone** (the `NodeID`â†’key mapper slot is permanent by
contract, so the id is never reused), and the read paths filter tombstoned
ids. Before v3.0.1 that tombstone lived only in memory, so it was lost the
moment the store was reopened from disk â€” the node came back as a live,
label-stripped row that no subsequent `DELETE` could remove. The defect was
acute for one-process-per-command callers that checkpoint and truncate the WAL
after every write (every delete was silently undone on the next command).

Three interlocking gaps were closed:

1. **Snapshot durability** â€” the checkpointer now writes the tombstone set to a
   new `tombstones.bin` snapshot component (magic + format version + sorted
   `uint64` ids + CRC32C, validated like every other component) before
   truncating the WAL.
2. **WAL-replay durability** â€” replaying an `OpRemoveNode` frame now
   reconstructs the tombstone (`g.RemoveNode`), in both the in-memory `txn`
   apply and recovery, so live and recovered state agree.
3. **Resurrection** â€” re-creating a removed key revives the node under the
   **same** stable `NodeID` (`lpg.Graph.AddNode` clears the tombstone), so a
   deleteâ†’recreate cycle yields exactly one live node â€” in-process, on WAL
   replay, and across a snapshot boundary.

On recovery the snapshot tombstone set is restored **after** the snapshot
nodes are materialised and **before** WAL replay, so any WAL-tail mutation
lands on top in chronological order.

### Recovery ordering (label/property freshness)

On the self-sufficient (`mapper.bin`) recovery path, the snapshot's labels and
properties are now applied **before** WAL replay. Previously they were
re-applied afterwards, so a WAL-tail delete-then-recreate could leave the node
carrying stale snapshot labels or clobber the re-created property values. WAL
deltas now win chronologically.

### Edge-deletion hygiene

Deleting an edge clears its per-pair edge labels and properties **once the
endpoint pair is fully disconnected** (no remaining parallel edge), so
re-creating an edge between the same endpoints no longer resurrects the removed
relationship's type or properties. The full-disconnect guard makes this
multigraph-safe: a shared per-pair label is retained until the last parallel
edge is removed. The Cypher relationship-delete path, the in-memory `txn`
apply, and WAL replay all route through `lpg.Graph.RemoveEdge`, so live and
recovered state agree.

### MERGE cross-type value equality (#1240)

The `MERGE` match phase now uses openCypher cross-type value equality, so
`MERGE (n:N {x: 1.0})` matches an existing node whose stored `x` is the integer
`1` instead of wrongly creating a duplicate node. This closed a TCK coverage
gap (the execution gate held at 3 897).

### `range()` int64 step overflow (#1238)

`range()` capped its element count overflow-safely but still materialised with
a comparison loop (`for i := start; i <= end; i += step`). For an under-cap
range whose last element sits at the `int64` boundary â€” e.g.
`range(9223372036854775805, 9223372036854775807)` â€” the final `i += step`
overflowed (`i` wrapped to `MinInt64`, the `i <= end` test stayed true), and
the loop marched across the entire `int64` space, appending unbounded elements
until the process hung and was OOM-killed. It now iterates exactly the
overflow-safe element count.

### Build tooling: standalone `staticcheck` (#1241)

The root `staticcheck.conf` used an invalid `[staticcheck "<path>"]` per-file
table that staticcheck's TOML parser rejects, aborting standalone
`staticcheck ./...` entirely (golangci-lint, the CI gate, was unaffected).
The config is repaired with directory-scoped overrides for the
documentation-only regexes in `cypher/tck/parser_only.go` and the ANTLR
generated parser, and a genuinely-dead test helper it surfaced was removed.
`staticcheck ./...` now exits 0 while keeping U1000 active across the rest of
the module.

## New (backward-compatible API)

This release adds public API in support of the durability fix. All additions
are backward-compatible:

- **`store/snapshot` â€” `tombstones.bin` component.** `WriteTombstones`,
  `ReadTombstones`, `ApplyTombstonesToGraph`, the `TombstonesReadback` type,
  the `TombstonesFile` constant, the `ErrTombstonesCorrupted` sentinel, and the
  `LoadedSnapshot.Tombstones` field. The component is **optional and additive**:
  it is emitted only when the graph has tombstoned nodes and does **not** bump
  the manifest version (it stays `3`), so snapshots are byte-identical for
  graphs that never delete and older snapshots load as an empty tombstone set.
- **`graph/lpg`** â€” `Graph.RemoveEdge` (the edge-deletion entry point that
  performs the per-pair hygiene above), `Graph.TombstonedIDs`,
  `Graph.TombstoneCount`, and `Graph.RestoreTombstones`.
- **`store/recovery`** â€” `Result.SnapshotTombstones` (count of tombstones
  restored from the snapshot before WAL replay).

> **Versioning note.** This release adds backward-compatible public API, which
> a strict reading of SemVer would classify as a *minor* bump. It is published
> as **v3.0.1** by maintainer decision because the dominant theme is durability
> bug-fixing and the additions exist only to make those fixes durable. See
> [docs/semver.md](../docs/semver.md).

## On-disk compatibility

- **Forward (upgrade): safe.** v3.0.1 reads every v1/v2/v3 snapshot and WAL
  produced by earlier releases; a snapshot without `tombstones.bin` loads with
  an empty tombstone set.
- **No format break.** The manifest version stays `3`; `tombstones.bin` is an
  optional component alongside `indexes/`. A graph that never deleted a node
  produces a byte-identical snapshot to v3.0.0.
- **Downgrade caveat.** A binary that predates `tombstones.bin` ignores the
  component, so reopening a v3.0.1-written store that contains tombstoned nodes
  with an older binary would resurrect those nodes. Upgrades are always safe.
