# Release Notes — GoGraph v2.0.0-rc2

**Date:** 2026-05-21
**Tag:** `v2.0.0-rc2`
**Commit:** `61573e6`

---

## Overview

v2.0.0-rc2 is the second release candidate for GoGraph v2.0.0. It builds
directly on [v2.0.0-rc1](v2.0.0-rc1.md) and delivers four sprint-worth of
improvements (sprints 37–40):

- **WAL durability** — all Cypher write operators now commit through the
  Write-Ahead Log, closing the major durability gap in rc1.
- **Parser conformance 99.5 %** — seven grammar gaps resolved (up from 90.7 %
  in rc1), with 380 additional integer-literal scenarios now correctly
  tokenised and executed.
- **Execution-level TCK 25.8 %** — UNWIND and variable-length expand wired,
  aggregation and ordering operators fully mapped; execution conformance more
  than doubled from 10.4 % in rc1.
- **Bolt benchmark suite** — three benchmark types at five concurrency levels
  published; both the 30-minute Cypher RW soak and the 1 024-connection 4-hour
  Bolt soak confirmed PASS with the gate threshold corrected.

This tag is a release candidate, not a stable release. v2.0.0 stable requires
≥ 80 % execution-level TCK. Current status: 25.8 %. See
[docs/semver.md](../docs/semver.md) for the full release-gate specification.

## What Changed Since rc1

### WAL Durability Bridge (Sprint 37)

All Cypher write operators (`CreateNode`, `CreateRelationship`, `Set`,
`Remove`, `Delete`, `DetachDelete`, `Merge`) now route through the WAL on
commit via the new `walMutatorAdapter`. This closes the "Cypher ↔ WAL
integration" known limitation listed in rc1 — properties and structural
changes set via `CREATE`, `SET`, and `MERGE` now survive a process restart.

A new `NewEngineWithStore` constructor accepts a `store.Store`, letting the
Cypher engine own its WAL-backed persistence lifecycle without requiring a
manual bridging step.

Four new persistence tests cover the full write-survive-restart round-trip:
WAL label survival, WAL property survival, crash simulation (recovery after
abrupt termination mid-write), and multi-snapshot accumulation.

### Parser Conformance: 90.7 % → 99.5 % (Sprints 38–39)

Seven grammar gaps resolved in sprint 38:

| Pattern | Example |
|---------|---------|
| Variable-length dotdot | `[*1..5]` |
| Zero-dot float literal | `.5` |
| Leading-dot float | `.123e4` |
| Negative hex/octal | `-0xFF` |
| Double NOT | `NOT NOT x` |
| Call without parentheses | `CALL db.labels` |
| Long float | `1.23e-45` |

Sprint 38 also adds multi-part query support: chained `WITH` clauses across
`MATCH … WITH … MATCH … RETURN …` patterns are now parsed and executed
correctly via a `MultiPartQ` rewrite.

In sprint 39, a bare-integer tokenisation bug was fixed: integer literals in
certain syntactic positions were mis-tokenised as identifiers, causing 380 TCK
scenarios to fail at parse time. The fix is transparent to callers.

### Execution Wiring: 10.4 % → 25.8 % (Sprint 39)

- `UNWIND list AS var` fully implemented (`Unwind` operator) and wired end-to-end.
- Variable-length path expand (`MATCH (a)-[*1..n]->(b)`) wired through
  the planner and execution engine.
- IR → exec operator mappings extended for all aggregation functions,
  `ORDER BY` expressions, and `SemiApply` / `RollUpApply` operators.
- Property access expressions (`n.prop`, `r.prop`) correctly resolve against
  the in-scope row record.
- `MATCH (n:L {prop: val})` now performs a hash-index seek (O(1) amortised)
  instead of a full label scan (O(|L|)).

### Bolt Benchmark Suite + Soak Confirmation (Sprint 40)

A new Bolt round-trip benchmark suite ships in `bench/soak`:

| Benchmark | Concurrency levels |
|-----------|-------------------|
| `BoltPingPong` | 1, 8, 64, 256, 1 024 goroutines |
| `BoltReadQuery` | 1, 8, 64, 256, 1 024 goroutines |
| `BoltWriteQuery` | 1, 8, 64, 256, 1 024 goroutines |

Results are published in `docs/benchmarks/cypher.md`.

The Bolt 4-hour gate threshold was corrected: the `MaxConnections` sentinel
had been set to the short-CI connection count rather than the full-run value,
causing false gate failures under `SOAK_FULL=1`. The fix is in the soak
harness only; no API or server change was required.

## Upgrade Guide

**From v2.0.0-rc1:** The only new required step is using `NewEngineWithStore`
if your application persists Cypher writes through the WAL. Callers that use
the in-memory engine (`NewEngine`) are unaffected. No API signatures changed.

**From v1.x:** See the rc1 upgrade guide — no additional breaking changes were
introduced in rc2.
