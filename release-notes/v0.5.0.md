# Release Notes — GoGraph v0.5.0

**Date:** 2026-06-17
**Tag:** `v0.5.0`
**Module:** `github.com/FlavioCFOliveira/GoGraph`
**Status:** Pre-1.0 (unstable public API)

## Overview

GoGraph v0.5.0 is the seventh published release of GoGraph, a Go module
for graph **persistence**, **manipulation**, and — above all — **fast
search**, designed to scale from in-memory graphs to graphs that exceed
RAM.

This is a pre-1.0 **MINOR** release. Its headline is **live `db.*`
schema introspection**: the built-in Cypher procedures `db.labels()`,
`db.relationshipTypes()`, and `db.propertyKeys()` now report only the
schema tokens **currently in use** — labels, relationship types, and
property keys attached to at least one live (non-tombstoned) node or
relationship — rather than every token ever indexed or seen. A token
leaves the result the moment its last bearing element is deleted.

The bump is **MINOR** under [Semantic Versioning](https://semver.org/):
the new behaviour is additive over `v0.4.0`, and three new exported
methods are added to `lpg.Graph`. The low-level helper
`procs.RegisterBuiltins` changed its third parameter from a single
closure to a `procs.BuiltinSources` struct — permitted in a `0.y.z`
line, and affecting only direct callers of that internal-style
registration helper, not users of the public Cypher engine. As a
`0.y.z` release the public API remains unstable: it may change without a
major-version bump while the module matures toward `1.0.0`. Pin the
exact version you depend on.

Both compliance guarantees that define the project continue to hold
without regression at this version:

- **100 % openCypher TCK-compliant at the execution level** — every
  scenario in `cypher/tck/features/` passes, with no `failed`,
  `undefined`, or `pending` step. The execution gate stands at
  **3 897 / 3 897 scenarios** and **16 006 / 16 006 steps**. The
  regression gate (`tckExecutionBaseline = 3897` in
  `cypher/tck/runner_test.go`) is unchanged. The `db.*` introspection
  procedures are **not** covered by the openCypher TCK.
- **100 % ACID-compliant** — Atomicity, Consistency, Isolation, and
  Durability hold across the in-memory engine and every persistence
  backend; the new introspection is **read-only** and never touches the
  write path.

## Highlights

- **Live `db.*` schema introspection.** `db.labels()`,
  `db.relationshipTypes()`, and `db.propertyKeys()` report only the
  schema tokens currently in use on live elements, filtering out
  tombstoned (deleted) nodes and relationships. A label, type, or key
  disappears from the result as soon as its last bearer is removed.
- **No more index-only labels.** `db.labels()` previously could report a
  label merely because an index existed for it; it now reports a label
  only while at least one live node bears it.
- **New `lpg.Graph` introspection API.** Three new exported methods —
  `NodeLabelsInUse`, `RelationshipTypesInUse`, and `PropertyKeysInUse` —
  expose the in-use enumeration directly on the graph type.

## What changed

### Live `db.*` schema introspection

The built-in `db.*` procedures now answer from live graph state, with
tombstone-aware filtering, instead of from index metadata or
never-collected token stores.

- **`db.labels()`** returns every distinct node label currently attached
  to at least one live node, whether or not an index exists for that
  label. The order is unspecified.

  ```cypher
  CALL db.labels()
  YIELD label
  RETURN label
  ```

  Yields: `label STRING`.

- **`db.relationshipTypes()`** returns every distinct relationship type
  currently attached to at least one live relationship. The order is
  unspecified.

  ```cypher
  CALL db.relationshipTypes()
  YIELD relationshipType
  RETURN relationshipType
  ```

  Yields: `relationshipType STRING`.

- **`db.propertyKeys()`** returns every distinct property key currently
  in use across nodes **and** relationships. The order is unspecified.

  ```cypher
  CALL db.propertyKeys()
  YIELD propertyKey
  RETURN propertyKey
  ```

  Yields: `propertyKey STRING`.

Each procedure is backed by a new enumeration method on the graph type:
`db.labels()` by `NodeLabelsInUse`, `db.relationshipTypes()` by
`RelationshipTypesInUse`, and `db.propertyKeys()` by `PropertyKeysInUse`.

### New `lpg.Graph` introspection methods

Three new exported methods are available directly on `lpg.Graph`:

```go
func (g *Graph[N, W]) NodeLabelsInUse() []string
func (g *Graph[N, W]) RelationshipTypesInUse() []string
func (g *Graph[N, W]) PropertyKeysInUse() []string
```

Each returns the distinct tokens currently in use, skipping tombstoned
elements. They are the single source of truth behind the `db.*`
procedures and can be called directly for Go-side schema introspection.

### `procs.RegisterBuiltins` signature change

The third parameter of `procs.RegisterBuiltins` is now a
`procs.BuiltinSources` struct that bundles the data-source closures the
built-in procedures query at invocation time:

```go
type BuiltinSources struct {
    ListConstraints   func() [][]expr.Value
    Labels            func() []string
    RelationshipTypes func() []string
    PropertyKeys      func() []string
}
```

Every field is optional; a nil closure makes its corresponding procedure
return an empty result set. This decouples the `procs` package from the
concrete graph type — the engine supplies pure closures that read live
graph state, and `procs` never imports the graph layer. Only direct
callers of `RegisterBuiltins` are affected; users of the public Cypher
engine are not.

### `db.schema.visualization()` — not yet implemented

`db.schema.visualization()` is registered but currently returns an empty
result set. It is documented for forward compatibility; do not rely on
its output until it is implemented.

## Divergence from Neo4j — `db.propertyKeys()`

Neo4j's `db.propertyKeys()` lists property-key tokens from a token store
that is never garbage-collected, so it keeps reporting a key even after
the last node or relationship using it is deleted. GoGraph instead
reports only the property keys **currently in use**: a key that no live
element bears is not listed.

This difference is observable, but it is **not** an
openCypher-conformance issue — the `db.*` introspection procedures are
outside the scope of the openCypher TCK, so GoGraph is free to choose
in-use semantics here.

## Upgrade notes

- **No migration required for engine users.** The on-disk format is
  unchanged from `v0.4.0`; stores written by earlier releases open
  without conversion. Cypher behaviour is unchanged except for the
  `db.*` introspection results described above.
- **If you call `procs.RegisterBuiltins` directly**, update the call
  site to pass a `procs.BuiltinSources` value instead of a single
  closure. Move the previous constraint-rows closure into the
  `ListConstraints` field; populate `Labels`, `RelationshipTypes`, and
  `PropertyKeys` as needed (any field may be left nil).
- **If you depend on the old `db.labels()` behaviour** that surfaced
  index-only labels, note that labels with no live bearer are no longer
  returned.

## Benchmark evidence summary

This release adds **read-only** schema introspection and changes no
hot-path data structure or search algorithm. There is therefore no
performance-relevant delta to report against `v0.4.0`; the headline
read-path benchmarks published in
[docs/benchmarks/v0.4.0.md](../docs/benchmarks/v0.4.0.md) remain
representative. The `db.*` procedures enumerate live tokens by scanning
graph state and are not on any query hot path.
