# Release Notes — GoGraph v0.2.0

**Date:** 2026-06-05
**Tag:** `v0.2.0`
**Module:** `github.com/FlavioCFOliveira/GoGraph`
**Status:** Pre-1.0 (unstable public API)

---

## Overview

GoGraph v0.2.0 is the second published release of GoGraph, a Go module
for graph **persistence**, **manipulation**, and — above all — **fast
search**, designed to scale from in-memory graphs to graphs that exceed
RAM.

This is a pre-1.0 **MINOR** release. It is **additive** over `v0.1.0`,
and its headline is a deep **reliability, ACID, and durability hardening
pass** across the persistence stack (`store/`), the Cypher engine
(`cypher/`), and the Bolt server (`bolt/`). The release adds true
explicit transactions over Bolt, makes Cypher constraints durable and
fully enforced, introduces a composed crash-safe `store.DB`, and bounds
every previously-unbounded result and import path so the module degrades
predictably under load instead of failing catastrophically.

Under [Semantic Versioning](https://semver.org/), a `0.y.z` version
signals that the public Go API is **not yet stable**: it may change
without a major-version bump while the module matures toward `1.0.0`.
Pin the exact version you depend on. This release tightens a small number
of behavioural defaults; read the [Upgrade notes](#upgrade-notes) before
upgrading.

Despite the pre-1.0 API label, the two compliance guarantees that define
the project continue to hold without regression at this version:

- **100 % openCypher TCK-compliant at the execution level** — every
  scenario in `cypher/tck/features/` passes, with no `failed`,
  `undefined`, or `pending` step. The execution gate stands at
  **3 897 / 3 897 scenarios** (16 006 / 16 006 steps). The regression
  gate (`tckExecutionBaseline = 3897` in `cypher/tck/runner_test.go`)
  rejects any change that lowers the count.
- **100 % ACID-compliant** — Atomicity, Consistency, Isolation, and
  Durability hold across the in-memory engine and every persistence
  backend. Durability is verified by the WAL recovery tests in
  `store/wal` and `store/recovery` and by the deterministic
  crash-injection battery in `internal/crashinject`.

Install with:

```bash
go get github.com/FlavioCFOliveira/GoGraph@v0.2.0
```

The Go module path is `github.com/FlavioCFOliveira/GoGraph` with **no**
`/vN` suffix, which is Semantic-Import-Versioning-correct for a `0.x`
line.

For the full, per-change log see [CHANGELOG.md](../CHANGELOG.md).

## Highlights

- **True explicit transactions over Bolt.** A Bolt transaction now owns
  a single engine transaction whose statements are **atomic together**,
  **isolated** by the single-writer serialisation, and **bounded** by a
  configurable timeout so the global write lock can never be held
  indefinitely. `ROLLBACK` now genuinely unwinds the transaction's
  writes.
- **Durable, fully enforced Cypher constraints.** `CREATE` / `DROP
  CONSTRAINT` survive a crash and a checkpoint, validate and seed against
  pre-existing data, and enforce `UNIQUE` for every property kind — not
  only strings.
- **A composed crash-safe store** (`store.DB`) that performs the one
  correct WAL-and-checkpoint teardown order on `Close`, removing a
  long-standing footgun for embedders.
- **Bounded resources end to end.** Result materialisation, aggregator
  buffers, the `Eager` pipeline-breaker, pattern comprehensions, the
  per-transaction WAL op buffer, and the import decoders all carry finite
  defaults now, so a pathological query or a crafted input fails fast
  with a typed error instead of exhausting memory.
- **Crash-durability ordering fixes** across the snapshot writer, the
  bulk loader, and the Cypher autocommit path, plus **fail-stop on
  genuine WAL corruption**.
- **Concurrency hardening**: a scoped-read CSR file API, a
  single-publisher generation contract, a leak-free checkpoint loop,
  context-aware blocking on the write path, and cancellation checks
  inside long traversals.
- **Security**: the Go toolchain is bumped to **go1.26.4**, resolving two
  reachable standard-library vulnerabilities; `govulncheck ./...` is
  clean.

## What is new

### Transactions and the Bolt server (`cypher/`, `bolt/`)

- **`Engine.BeginTx` → `*ExplicitTx`** with `Exec` / `ExecAny` /
  `Commit` / `Rollback`, wired through the Bolt `BEGIN` / `RUN` /
  `COMMIT` / `ROLLBACK` protocol on both the WAL-backed and store-less
  engine wirings. Writes accumulate eagerly and are made durable and
  visible as one unit on `COMMIT` (the WAL is `fsync`ed once), or unwound
  together on `ROLLBACK`. Write-write isolation is enforced by the store
  single-writer mutex (WAL-backed) or a dedicated engine write mutex
  (store-less), held for the lifetime of the transaction.
- **Bounded transaction lifetime.** The transaction is rooted at the
  connection context with a configurable `Server.Options.DefaultTxTimeout`
  (default 30 s), and the connection handler rolls the transaction back
  on every teardown path — disconnect, error, panic, or `FAILED` — so the
  global write lock is released promptly rather than at garbage
  collection.
- **Typed result-cap failures.** The Bolt server maps the engine's
  bounded-resource sentinels to `Neo.ClientError.General.LimitExceeded`,
  so an over-broad `RUN` is rejected cleanly and the connection recovers
  with a `RESET`. `Engine.ResultRowCap()` exposes the configured cap, and
  `NewServer` logs a loud warning when handed a deliberately uncapped
  engine.
- **Server-level metrics.** Six counters are emitted through the internal
  metrics backend: `bolt.server.conn.accepted` / `.rejected` /
  `.closed`, and `bolt.server.tx.opened` / `.closed` / `.abandoned`. The
  live-connection and open-transaction values are derived as paired
  counters (`live = opened − closed`).

### Cypher constraints (`cypher/`, `store/`)

- `CREATE CONSTRAINT` / `DROP CONSTRAINT` are routed through a typed WAL
  operation, replayed on recovery, and carried in a `constraints.bin`
  snapshot component so they survive a checkpoint and WAL truncation.
- `CREATE CONSTRAINT` scans the labelled nodes, **rejects** creation over
  already-duplicated (`UNIQUE`) or null (`NOT NULL`) pre-existing data,
  and seeds the value-set so subsequent writes are validated against
  pre-existing values.
- The canonical value-set key covers every property kind (`float` via its
  IEEE-754 bits, `bool`, `time` as RFC 3339, `[]byte` as base64, list),
  so a `UNIQUE` constraint on a non-string property is enforced instead
  of being a silent no-op.
- `NewEngineWithStoreAndConstraints` re-registers recovered constraints
  on open.

### Storage and persistence (`store/`)

- **`store.DB`** — a new top-level package with a composed `DB` that owns
  the WAL and an optional checkpointer and runs the crash-safe teardown
  order in `Close` / `CloseCtx`: optional final checkpoint → stop the
  checkpoint goroutine → close the WAL. It is idempotent and safe under
  concurrent `Close`. `bolt.Server` adopts it via an optional
  `Options.Closer`.
- **Commit-serialised checkpointing** — `txn.Store.RunUnderCommitLock`
  and `checkpoint.WithCommitSerialiser` run the snapshot-and-truncate
  window under the store's real commit lock, so a checkpoint can never
  capture a partial transaction or truncate away a transaction committed
  during the snapshot.
- **Scoped-read CSR file API** — `csrfile.Reader.Read` holds an internal
  read lock for the duration of a callback, so `Close` blocks until every
  in-flight reader returns before it unmaps the `mmap` region. The
  semi-external `search/extern` BFS and PageRank traversals run inside
  this scope; their results are byte-for-byte unchanged.
- **Persisted graph configuration** — the snapshot manifest now records
  the graph's directed / multigraph shape, so a graph created simple no
  longer silently becomes a multigraph after a reopen. Manifests without
  the field default to the previous behaviour, so every existing snapshot
  replays unchanged.

### Graph model (`graph/lpg`, `graph/adjlist`)

- New accessors: `lpg.Graph.FirstEdgeHandle` (the stable handle of the
  first `src → dst` slot), `lpg.Graph.Config` and `adjlist.AdjList.Config`
  (the immutable directed / multigraph configuration), and
  `lpg.Graph.ValidateNode` (run the installed schema validator's
  whole-node check at finalisation).

## What changed (behavioural)

This release tightens three categories of default behaviour. Each change
makes the module safer under load; each is summarised here and detailed
in the [Upgrade notes](#upgrade-notes).

- **Finite default resource caps** now apply where results were
  previously unbounded (result rows, aggregator element budgets, the
  `Eager` pipeline-breaker, pattern comprehensions, and the
  per-transaction WAL op buffer).
- **Import decoders enforce a byte ceiling** and now return a **nil**
  graph on **any** decode error (uniform all-or-nothing import).
- **Re-entrant `Graph.View` / `ApplyAtomically` now panics** instead of
  deadlocking.

## Reliability, ACID, and durability hardening

The bulk of v0.2.0 is correctness work that strengthens the project's
non-negotiable invariants. The most consequential fixes:

- **Durable-before-visible writes.** Cypher autocommit mutations are made
  durable (WAL `fsync`) **before** they become visible to concurrent
  readers, so a reader can no longer observe a write that a crash would
  then lose.
- **Crash-safe `fsync` ordering.** The snapshot writer `fsync`s the
  staging directory before the publish rename; the bulk loader `fsync`s
  the parent directory after its rename. The crash-safe ordering is now
  write-and-`fsync` each component → `fsync` the directory → rename →
  `fsync` the parent.
- **Fail-stop on genuine WAL corruption.** Recovery returns genuine
  corruption (CRC mismatch, unsupported record version) to the caller
  instead of swallowing it, and the shipped examples refuse to append to
  a corrupt WAL. A benign torn or truncated tail remains a clean success;
  `Result.IsClean` reports the distinction.
- **Atomic in-memory rollback.** A Cypher write query that errors or
  panics mid-execution rolls its in-memory mutations back through an
  inverse-op undo log inside the visibility barrier, so the in-memory
  graph never diverges from the durable state — including restoring
  per-handle edge identity on the multigraph removal-then-fail
  interleaving.
- **Context-aware blocking and cancellation.** The engine's write-path
  store acquisition uses a cancellable semaphore, and five long
  traversals (direction-optimised BFS, diameter, Tarjan SCC, triangle
  counting, and Cypher `DETACH DELETE`) poll for cancellation inside
  their inner loops.
- **Bounded resources, never panic.** Per-transaction, per-group, and
  `Eager` buffers are bounded; the `Distinct` cap counts distinct rows
  rather than hash buckets; and a coarse result-byte budget complements
  the row cap so a single very wide row cannot exhaust memory.
- **Concurrency-primitive hardening.** The scoped-read CSR API prevents a
  use-after-unmap; the `generation` publisher enforces its
  single-publisher contract and fixes a missed-wakeup; the checkpoint
  loop drains its trigger channel on exit so a `Trigger` racing `Stop`
  cannot leak a goroutine; the Bolt session rolls its transaction back at
  the `FAILED` transition; and the metrics hot path is nil-safe.
- **Integer-overflow guard in network flow.** Capacity and cost
  accumulation validate their inputs and return `ErrCapacityOverflow`
  instead of silently producing a wrapped, negative result.
- **Schema required-property enforcement.** A declared required property
  is now enforced at node finalisation via `Graph.ValidateNode`, instead
  of being advisory-only.

## Security

The Go toolchain is bumped to **go1.26.4** (`chore(toolchain)` commit
`6969201`), resolving two Go standard-library vulnerabilities reachable
from the module's own code paths:

| Advisory | Package | Reached via |
|---|---|---|
| **GO-2026-5039** | `net/textproto` | `snapshot.Open` → `textproto.Reader.ReadMIMEHeader` |
| **GO-2026-5037** | `crypto/x509` | the Bolt TLS handshake (`bolt/proto/handshake.go`) |

Only the `toolchain` directive changed; the `go 1.26` language directive
is left untouched (no language-version bump, no new features adopted).
`govulncheck ./...` reported two affecting vulnerabilities before the bump
and **none** afterwards. CI consumes the toolchain via
`go-version-file: go.mod`, so the single edit propagates the bump to every
job.

## Performance

The one performance change in this release is a memory-footprint
improvement to **Personalised PageRank** (`search/centrality`): the
push-relabel worklist now compacts its consumed prefix to track the live
frontier, bounding the worklist's backing array to the frontier size
rather than letting it grow toward the total step count. The rank vector
is **bit-for-bit identical** to the previous implementation.

The headline single-threaded benchmark figures are unchanged from
[v0.1.0](v0.1.0.md#performance); re-measure on your own hardware before
relying on any absolute number. Several hardening fixes were validated to
be allocation-neutral with `benchstat`; the one documented cost is a
small per-row lookup floor introduced by the optional result-byte budget,
which is the irreducible price of the bounded-resource guarantee.

## Upgrade notes

This release is **API-additive** over `v0.1.0`: no exported identifier
was removed or renamed. The following **behavioural** changes may affect
existing callers and should be reviewed before upgrading.

1. **Finite default result and buffer caps.** Result materialisation,
   per-group aggregators, the `Eager` pipeline-breaker, pattern
   comprehensions, and the per-transaction WAL op buffer now apply finite
   defaults where they were previously unbounded:
   - `DefaultMaxResultRows`, `DefaultMaxCollectItems`, and
     `DefaultMaxEagerRows` are `10_000_000`.
   - `DefaultMaxGroups` is `1_000_000`.
   - `DefaultMaxResultBytes` is 1 GiB.
   - `DefaultMaxTxnOps` is `16_000_000`.

   A query or transaction that exceeds a default now returns a typed
   bounded-resource error (surfaced through `Result.Err()`, and mapped to
   `Neo.ClientError.General.LimitExceeded` over Bolt). The defaults are
   high enough that the openCypher TCK and every shipped example stay
   green. If you deliberately need an unbounded result, set the
   corresponding `EngineOptions` field to its unbounded sentinel
   (`MaxResultRowsUnlimited`, `MaxResultBytesUnlimited`, or
   `MaxCollectItemsUnlimited`, all `-1`).

2. **Import byte ceiling and nil-graph-on-error.** The CSV, JSON Lines,
   and GraphML decoders now reject input larger than `DefaultMaxBytes`
   (1 GiB) with the typed `ErrInputTooLarge` (configurable; `MaxBytes <= 0`
   means unlimited), and they now return a **nil** graph on **any** decode
   error — a parse error, context cancellation, or the byte-ceiling
   error. If your code previously inspected the partially-populated graph
   returned alongside an error, switch to treating any error as a total
   failure; the typed error values are unchanged.

3. **Re-entrant `View` / `ApplyAtomically` now panics.** A goroutine
   already inside the visibility barrier that nests another
   `lpg.Graph.View` or `ApplyAtomically` call now panics with an
   actionable message instead of silently deadlocking. Production code
   never nests these calls; if yours does, restructure it so the inner
   read or write runs outside the enclosing barrier. The panic is
   intentional and is never recovered, because it indicates a programmer
   error.

Because the public API is pre-1.0, continue to **pin the exact version**
you depend on.
