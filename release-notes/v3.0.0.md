# Release Notes — GoGraph v3.0.0

**Date:** 2026-06-01
**Tag:** `v3.0.0`
**Module:** `github.com/FlavioCFOliveira/GoGraph`
**Status:** Stable

---

## Overview

GoGraph v3.0.0 is a **security-hardening and distribution** release. It
delivers a complete, proof-of-concept-confirmed remediation of a 20-finding
security audit and adopts the conventional URL-based Go module path so the
library can be consumed directly with `go get`. The two compliance guarantees
that define the project were held on every commit:

- **100 % openCypher TCK-compliant at the execution level** — the gate stayed
  at **3 897/3 897 scenarios** (16 006/16 006 steps) throughout the audit
  remediation and the module rename, with no `failed`, `undefined`, or
  `pending` steps. The regression gate (`tckExecutionBaseline = 3897` in
  `cypher/tck/runner_test.go`) rejects any pull request that lowers the count.
- **100 % ACID-compliant** — Atomicity, Consistency, Isolation, and Durability
  hold across the in-memory engine and every persistence backend. The
  highest-impact security fix on the transactional path (H7, panic boundaries)
  was specifically designed to strengthen, never weaken, the ACID invariants.

For the full, per-finding change log see [CHANGELOG.md](../CHANGELOG.md). To
upgrade an existing 2.x deployment, follow the
[2.x → 3.x migration guide](../docs/migration-2-to-3.md).

## Breaking changes

v3.0.0 keeps the breaking surface deliberately narrow — two changes, both
mechanical to adopt. They are listed first because every consumer must act on
them.

### 1. Module import path

The Go module path changed from `gograph` to
**`github.com/FlavioCFOliveira/GoGraph`** (#1239). Every importer must rewrite:

```go
// Before (v2.x)
import "gograph/graph/lpg"
import "gograph/cypher"

// After (v3.0.0)
import "github.com/FlavioCFOliveira/GoGraph/graph/lpg"
import "github.com/FlavioCFOliveira/GoGraph/cypher"
```

This adopts the conventional, repository-matching path so the module can be
fetched with `go get github.com/FlavioCFOliveira/GoGraph`. The rename rewrote
the `go.mod` module line and all internal import paths across the tree,
updated godoc doclinks and usage documentation, and adjusted the
`goimports` local-prefix and the coverage-gate regexes.

**Unchanged on purpose:** the `gograph` package identifier (your code still
writes `lpg.New(...)`, `cypher.NewEngine(...)`, and so on — only the import
*path* changes), the `gograph-*` cache-directory names, the `rmp` roadmap
name, and the frozen historical benchmark captures.

**No `/v3` path suffix.** The import base stays
`github.com/FlavioCFOliveira/GoGraph` with no major-version path suffix. This
is a deliberate decision that matches the v2.0.0 precedent, which was likewise
tagged without a major-version path suffix.

### 2. `bolt/server.NewServer` now returns an error and fails closed

```go
// Before (v2.x): a nil Auth silently installed NoAuthHandler — Options{}
// shipped a fully open server.
func NewServer(eng *cypher.Engine, opts Options) *Server

// After (v3.0.0): NewServer fails closed; a nil Auth is an error.
func NewServer(eng *cypher.Engine, opts Options) (*Server, error)
```

`NewServer` is now **secure by default**. A nil `Options.Auth` returns the new
`ErrNoAuthHandler` sentinel instead of installing the open-door handler. To run
without authentication — only for development or testing — set
`Auth: NoAuthHandler{}` explicitly; the explicit value is itself the opt-in and
still emits the loud insecure-default warning at startup:

```go
srv, err := server.NewServer(eng, server.Options{
    Auth: server.NoAuthHandler{}, // explicit opt-in to no-auth (dev/test only)
})
if err != nil {
    return err
}
```

This change ships finding L1 of the security audit (see below). The
step-by-step rewrite, including the error-handling diff, is in the
[2.x → 3.x migration guide](../docs/migration-2-to-3.md).

## The 2026 security audit

The headline of v3.0.0 is a structured security audit and its complete
remediation, planned and executed across sprints **S140–S143** of the
`gograph` roadmap. The audit produced **20 findings**; all 20 are fixed in this
release. Each fix was confirmed with a proof of concept, landed under the
project's `Specify → Implement → Test → Document` workflow, and was validated
against the two compliance invariants: the openCypher TCK execution gate held
at **3 897/3 897** and no ACID property was weakened.

### Findings by severity

| Severity | Count | Findings |
|----------|-------|----------|
| Critical | 2     | C1, C2 |
| High     | 7     | H1, H2, H3, H4, H5, H6, H7 |
| Medium   | 1     | M1 |
| Low      | 4     | L1, L2, L3, L4 |
| Info     | 6     | I1, I2, I3, I4, I5, I6 |
| **Total**| **20**| |

### Critical

- **C1 — pre-auth stack-overflow DoS in the PackStream decoder.**
  `bolt/packstream.Decoder.ReadValue` recursed one stack frame per
  List/Map/Struct nesting level with no bound. A single pre-authentication
  `HELLO` frame of repeated `0x91` markers triggered a *fatal*
  (unrecoverable) stack overflow that killed the whole process. Bounded to
  `maxValueDepth = 128` with the `ErrNestingTooDeep` sentinel across all three
  recursive arms; well-formed nested values up to the bound still decode
  (#1218).
- **C2 — stack-overflow DoS from an untrusted Cypher query.** The parser's
  AST visitor recursed one frame per nesting level with no bound. A ~300 KB
  query of nested brackets triggered a fatal stack overflow reachable from
  `Engine.Run` and Bolt `RunInTxAny`. `guardInput()` now runs as the first
  statement of `Parse`/`ParseStrict`, enforcing `maxQueryBytes = 1<<20` and
  `maxNestingDepth = 256`; the depth check is an O(n) scanner that correctly
  ignores brackets inside string literals and comments. It returns a
  `*ParseError`, never crashes (#1219).

### High

- **H1 — OOB read via crafted CSR-file header.** `store/csrfile.bindSlices`
  reinterpreted mmap bytes using counts/offsets read straight from the header
  with no validation; the file's CRC32C is integrity-only, so a crafted header
  with a recomputed CRC could panic on a bounds check or — via `uint64`
  overflow of `8 * NEdges` wrapping in-bounds — build an `unsafe.Slice` view
  far past the mmap (out-of-bounds read / disclosure). `Header.validate` now
  recomputes the canonical overflow-safe layout and requires exact offset
  equality before the slice, returning `ErrFileCorrupted` (#1226).
- **H2 — OOM DoS via PackStream length prefix.** The decoder allocated a
  slice/map sized by a wire-controlled 32-bit length prefix *before* reading
  the bytes, so a 5-byte marker could force a multi-gigabyte (bytes/string) or
  ~64 GiB (list/map) allocation pre-auth. A per-message remaining-byte budget,
  seeded from the source length, now rejects any length/count exceeding the
  input with `ErrLengthExceedsInput` before the allocation; unknown-length
  readers fall back to a 16 MiB cap (#1221).
- **H3 — OOM DoS via snapshot CSR manifest.** `store/snapshot.ReadCSR`
  allocated slices sized by unvalidated `uint64` counts from the file
  (gigabytes for `nV = 1<<28`, terabytes near `2^40`) and computed weight bytes
  with an overflowing multiply. `readCSRLimited` now bounds every real load
  path (`Open`, `LoadSnapshotFull`) by the manifest `FileEntry.Size`, with a
  `1<<40` backstop and overflow-safe (`bits.Mul64`) weight sizing (#1227).
- **H4 — OOB read via crafted CSR vertex-offset array.**
  `store/snapshot.ApplyCSRToGraph` read vertex and edge offsets from the file
  with no check that the offset array was monotonic or in range, so a crafted
  offset (e.g. `1<<40`) drove an out-of-bounds panic and a `start > end` pair
  underflowed a metric counter. The offset array is validated once
  (`Vertices[i] <= Vertices[i+1]` and `Vertices[last] <= len(Edges)`) before
  replay, returning `ErrCorrupted` (#1228).
- **H5 — path traversal via snapshot manifest index name.**
  `store/snapshot.LoadIndexes`/`WriteIndexes` built
  `filepath.Join(idxDir, name+".bin")` from the manifest's attacker-controllable
  index name, so `"../../../../tmp/pwned"` or an absolute path escaped `idxDir`
  — an arbitrary file read on the recovery path. `validateIndexName` now
  rejects any name that is `""`/`"."`/`".."`, is not equal to its own
  `filepath.Base`, fails `fs.ValidPath`, contains a path separator or a `".."`
  substring, or is absolute, failing stop with `ErrManifestCorrupted` (#1229).
- **H6 — OOM / `makeslice` panic via `range()`.** `cypher/funcs.range()`
  pre-computed a capacity as `(end-start)/step + 1`, which could overflow or be
  astronomically large: `range(1, 9223372036854775807)` panicked with
  `makeslice: cap out of range`, and `range(1, 5000000000)` allocated tens of
  gigabytes and was OOM-killed — both reachable from any untrusted query. The
  materialised element count is capped at `maxRangeElements = 1e8` (far above
  the largest TCK bound), derived overflow-safely in `uint64`, returning a
  typed `expr.EvalError` (`ArgumentError: NumberOutOfRange`) over the cap
  (#1235).
- **H7 — panic-driven process kill on the Bolt and engine paths.** A
  recoverable panic in the Bolt per-connection goroutine or in
  `Engine.Run`/`RunInTx` would unwind past the library and crash the whole
  process. `recover()` boundaries were added in `handleConn` (log +
  `bolt.server.conn.panics` metric + close one connection) and in
  `Run`/`RunInTx` (`ErrInternalPanic` typed error + log + panics metric). The
  `RunInTx` boundary also **rolls back the in-flight WAL transaction**, so the
  store single-writer mutex is never leaked on a panic — an ACID-preserving
  fix, since a subsequent write no longer deadlocks. A deadlock-freedom
  regression test covers the write path. This does not affect fatal stack
  overflows, which are handled by C1/C2 (#1220).

### Medium

- **M1 — Slowloris connection-slot exhaustion.** `bolt/server.Options.ConnTimeout`
  defaulted to `0`, so no I/O deadline was ever applied and the handshake could
  block forever; an attacker could hold every connection slot with one byte
  each. `NewServer` now defaults `ConnTimeout` to 30 s (idle, post-auth), and
  the unauthenticated handshake is bounded by a 10 s `DefaultHandshakeTimeout`
  (set before `Negotiate`, cleared on success). `Options` is unchanged in size,
  so this is not an API break beyond the `NewServer` return-value change (L1)
  (#1222).

### Low

- **L1 — open-by-default Bolt server (BREAKING).** `NewServer` silently
  installed `NoAuthHandler` when `Options.Auth` was nil, so `Options{}` shipped
  a fully open server. `NewServer` now returns `(*Server, error)` and rejects a
  nil `Auth` with `ErrNoAuthHandler`; running without authentication requires
  explicitly setting `Auth: NoAuthHandler{}` (which still logs the loud
  warning). See [Breaking changes](#breaking-changes) above and the migration
  guide (#1223).
- **L2 — world-readable persistence files.** The WAL, CSR files, and snapshot
  component files were created world-/group-readable (`0o644`, or
  `0o666 & ~umask`), exposing full graph data. They are now created `0o600`
  (owner-only); the atomic write+rename publish protocol and all
  fsync/durability semantics are unchanged (#1230).
- **L3 — no transport hardening baseline.** The server applied no
  `MinVersion`/cipher policy to an operator-supplied `tls.Config`. The new
  exported `DefaultTLSConfig()` returns a fresh hardened baseline (TLS 1.2
  floor, modern AEAD/ECDHE cipher suites, TLS 1.3 auto-negotiated) for
  operators to start from. Behaviour is unchanged for callers that pass their
  own config; a nil `TLSConfig` still means plaintext (#1224).
- **L4 — pre-auth ROUTE and leaky decode errors.** `ROUTE` was accepted in
  `StateNegotiation`, letting an unauthenticated client elicit a routing-table
  response; it is now accepted only from `StateReady`/`StateTxReady` (verified
  compatible with `neo4j-go-driver`, which always completes `HELLO`/`LOGON`
  first). The serve-loop decode-error `FAILURE` message now goes through
  `sanitiseErr` instead of returning the raw internal error string, while still
  being logged server-side (#1225).

### Info (defence-in-depth)

- **I1** — `store/csrfile.Reinterpret`: overflow-safe `size*n` sizing via
  `math/bits.Mul64`; an overflowing product saturates to `math.MaxInt` so the
  call takes the existing "data too short" panic path rather than producing an
  out-of-bounds alias (#1231).
- **I2** — `store/wal.Decode`: a `maxFrameSize = 1<<30` (1 GiB) bound rejects
  an oversized `plen` with `ErrFrameTooLarge` before the allocation, layered
  over the existing `uint32` bound (#1232).
- **I3** — `store/snapshot` and `store/recovery`: list-property decoders clamp
  the capacity hint to `min(count, remaining/minElemBytes)`, bounding a hostile
  count while pre-sizing legitimate lists exactly (#1233).
- **I4** — `store/snapshot`: component reads use `O_NOFOLLOW` on Unix
  (`openSnapshotComponent`), so a symlinked component in an untrusted snapshot
  directory is rejected with `ELOOP` rather than dereferenced; degrades to a
  plain open on Windows (#1234).
- **I5** — `cypher`: `Engine.Run`/`RunInTx` check the context before
  `parseAndAnalyse` / any `store.Begin`, so a cancelled context interrupts an
  expensive parse promptly (#1236).
- **I6** — `cypher/expr`: a bounded, concurrency-safe FIFO cache (capacity
  1 024) of compiled `*regexp.Regexp` for the `=~` operator removes per-row
  recompilation; behaviour is byte-identical, including the
  NULL-on-invalid-pattern mapping, and the bound prevents an untrusted pattern
  from driving unbounded growth (#1237). See also
  [Performance](#performance).

## Module rename rationale

Through v2.0.0 the module declared the short path `gograph`, with the
documented import base `github.com/xumiga/gograph`. That short path could not
be fetched with `go get`, which expects a module path that resolves to a
version-control repository. v3.0.0 adopts
**`github.com/FlavioCFOliveira/GoGraph`** — the path that matches the public
GitHub repository — so the module is fetchable and importable by external
consumers in the conventional way.

The rename touched the `go.mod` module line and every internal import path
(across the whole tree), godoc doclinks, the usage documentation, the
`goimports` local-prefix in `.golangci.yml`, and the coverage-gate regexes;
import groups were re-sorted accordingly. It is purely a path change: the
`gograph` package identifier and every exported symbol are unchanged, so the
adopter's work is a mechanical search-and-replace on import paths (see the
migration guide). No `/v3` major-version path suffix is added, matching the
v2.0.0 precedent.

## Performance

- **`cypher/expr` `=~` operator (I6, #1237).** The regex operator previously
  recompiled its pattern on every evaluated row via `regexp.MatchString`, so
  `WHERE x =~ $p` over a large scan recompiled the same pattern once per row.
  (Go's `regexp` is RE2, so this was redundant-compilation overhead, not
  ReDoS.) Each distinct pattern is now compiled at most once via a bounded
  FIFO cache, removing the redundant compilation from the hot path. Match
  results and the NULL-on-invalid-pattern mapping are unchanged, and the cache
  bound prevents a hostile pattern stream from growing memory without limit.

No algorithm, persistence-format, or hot-path data-structure change ships in
v3.0.0, so the benchmark figures and methodology in
[`docs/benchmarks/cypher.md`](../docs/benchmarks/cypher.md) carry forward from
v2.0.0.

## Validation evidence

All checks were run at the release commit on Go 1.26.3, darwin/arm64. The TCK
runner pins `ZONEINFO` to `cypher/tck/testdata/zoneinfo-slim.zip` for
deterministic temporal conformance across hosts.

| Suite | Scope | Result |
|-------|-------|--------|
| `go build ./...` | Compilation | OK |
| `go vet ./...` | Static analysis | OK |
| `go mod tidy` + `go mod verify` | Dependency integrity | Clean (no `go.mod`/`go.sum` delta); all modules verified |
| **`TestTCKExecution`** | **3 897 scenarios / 16 006 steps** | **3 897 passed, 16 006 passed, 0 undefined (baseline = 3 897)** |
| `go test -race ./...` | Race detector, all packages | PASS — 87 ok, 0 FAIL, 0 data races, 4 no-test packages |
| `golangci-lint run ./...` (v2.12.2) | Lint | 0 issues |
| `govulncheck ./...` | Vulnerability scan | No vulnerabilities found |
| Coverage gate | Aggregate ≥ 85 %, per-package ≥ 75 % | 86.0 % aggregate — PASS |

**Key invariants verified:** 100 % execution-level openCypher conformance
(3 897/3 897) held across the entire audit remediation and the module rename;
ACID atomicity, consistency, isolation, and durability preserved across the
in-memory engine and every persistence backend (the H7 panic-boundary fix
strengthens isolation/durability by rolling back the in-flight WAL transaction
and never leaking the single-writer mutex); zero data races; zero goroutine
leaks; bounded resources on every queue, pool, and cache.

> **Note on the known timing flake.** The `bolt/server`
> `TestE2E_FailureTimeout` end-to-end test can flake under heavy parallel
> `-race` load (it passes in isolation and exhibits no data race). It did not
> trigger in the release validation run above; it is a pre-existing,
> non-blocking timing artefact, not a correctness defect.

## Migrating from 2.x to 3.0

The authoritative, step-by-step walkthrough — with copy-pasteable diffs — is in
[`docs/migration-2-to-3.md`](../docs/migration-2-to-3.md). In short:

1. **Rewrite import paths.** Replace every `import "gograph/..."` with
   `import "github.com/FlavioCFOliveira/GoGraph/..."`. The package identifiers
   in your code do not change.
2. **Update `bolt/server.NewServer` call sites.** Handle the new `error`
   return, and pass `Auth: NoAuthHandler{}` explicitly if you were relying on
   the old implicit no-auth default (development/testing only).
3. **Re-run `go mod tidy && go mod verify`** and rebuild.

There are no on-disk format changes in v3.0.0; existing WAL, snapshot, and CSR
files load unchanged.
