# Release Notes — GoGraph v0.3.1

**Date:** 2026-06-15
**Tag:** `v0.3.1`
**Module:** `github.com/FlavioCFOliveira/GoGraph`
**Status:** Pre-1.0 (unstable public API)

---

## Overview

GoGraph v0.3.1 is the fourth published release of GoGraph, a Go module
for graph **persistence**, **manipulation**, and — above all — **fast
search**, designed to scale from in-memory graphs to graphs that exceed
RAM.

This is a pre-1.0 **PATCH** release and a **performance and
security-hardening release**. It is **API-additive** over `v0.3.0`: no
exported identifier was added, removed, or renamed, there is **no
breaking change**, and there is **no new user-facing public API**. The
performance work is correctness-preserving by construction — the new
write and analytics paths produce **byte-identical / bit-identical**
results and regress **nothing** on the curated benchmark set — which is
precisely why this qualifies as a patch rather than a minor release.

The release consolidates two streams of work:

- a focused **performance cycle** (rmp sprints S-PA1..S-PA6, tasks
  #1497–#1525) that lifts the write, analytics, query, and read paths
  with measured, benchstat-gated wins; and
- two further **security audits** — SEC-2026-06-14b (9 findings,
  #1480–#1489) and SEC-2026-06-14c (7 findings, #1490–#1496) — both fully
  remediated, each shipping a strict regression battery.

Under [Semantic Versioning](https://semver.org/), a `0.y.z` version
signals that the public Go API is **not yet stable**: it may change
without a major-version bump while the module matures toward `1.0.0`.
Pin the exact version you depend on. This release introduces **no
breaking change and no behavioural change** to any documented API
contract; the [Upgrade notes](#upgrade-notes) are therefore short.

Both compliance guarantees that define the project continue to hold
without regression at this version:

- **100 % openCypher TCK-compliant at the execution level** — every
  scenario in `cypher/tck/features/` passes, with no `failed`,
  `undefined`, or `pending` step. The execution gate stands at
  **3 897 / 3 897 scenarios** (error-fidelity gate 121). The regression
  gate (`tckExecutionBaseline = 3897` in `cypher/tck/runner_test.go`)
  rejects any change that lowers the count.
- **100 % ACID-compliant** — Atomicity, Consistency, Isolation, and
  Durability hold across the in-memory engine and every persistence
  backend. The new group-commit write path was **certified by the
  storage-engine auditor** as preserving all four properties; durability
  is verified by the WAL recovery tests in `store/wal` and
  `store/recovery` and by the deterministic crash-injection battery in
  `internal/crashinject`.

Install with:

```bash
go get github.com/FlavioCFOliveira/GoGraph@v0.3.1
```

The Go module path is `github.com/FlavioCFOliveira/GoGraph` with **no**
`/vN` suffix, which is Semantic-Import-Versioning-correct for a `0.x`
line. The Go toolchain remains **go1.26.4** (unchanged from `v0.3.0`).

For the full, per-change log see [CHANGELOG.md](../CHANGELOG.md).

## Highlights

- **~118× concurrent write throughput** from group commit / WAL fsync
  coalescing, with **zero single-thread regression** and full ACID
  certification.
- **1.7–2.4× PageRank** on large graphs from a parallel pull-formulation
  over a reverse-CSR, with **bit-for-bit identical** results regardless
  of `GOMAXPROCS`.
- **~114× range-predicate index seek** on selective indexed ranges, with
  **zero regression** on the curated benchmark set.
- **Hash join** for disconnected equi-join patterns (≈ 93× faster on the
  targeted shape), with an **identical result multiset** proven against
  the nested-loop path.
- A **real B+ tree** replaces the sorted-array property index, and a
  Cypher-created `btree` index is now bound, backfilled, and
  self-maintaining.
- **Cypher read-path allocation cuts** (column-oriented result rows,
  deferred node materialisation, shared property/label views, and a
  lock-free metadata name registry).
- **Two security audits closed** — 9 + 7 findings remediated, with no
  Critical finding and no prior-fix regression.

## What changed (behavioural)

This release introduces **no breaking API change and no behavioural
change** to any documented API contract. Every performance improvement
is correctness-preserving:

- the group-commit write path leaves the on-disk WAL stream
  byte-identical and recovery unchanged (all-or-nothing on the commit
  marker);
- parallel PageRank produces ranks that are bit-for-bit identical to the
  serial path for every vertex, independent of `GOMAXPROCS` or
  scheduling;
- the range-index seek and hash join are plan-build physical
  substitutions, each proven against a differential test to return an
  identical result multiset to the path it replaces.

One additive, **opt-in** surface appears as a side effect of a security
fix: a disconnected multi-pattern `MATCH` now surfaces a Cartesian-product
**notification** through the new `Result.Notifications()` accessor and the
Bolt PULL `SUCCESS` metadata (#1483). This is purely informational — it
changes no result and breaks no existing caller, which simply ignores the
new metadata.

## Performance

The 2026-06-14 performance cycle (rmp sprints S-PA1..S-PA6, tasks
#1497–#1525) is the bulk of this release. Every change is benchstat-gated
against the `f6f8c7a` baseline (ledger row 0006), the curated guard band
(Dijkstra / BFS / Brandes) stays flat throughout, and each item below is
faithful to a committed, measured change. The raw before/after numbers
live in [docs/benchmarks/history/](../docs/benchmarks/history/); the
per-release report is [docs/benchmarks/v0.3.1.md](../docs/benchmarks/v0.3.1.md).

### Group commit / WAL fsync coalescing (`store/`, #1507)

The per-commit `fsync` was the dominant write-throughput ceiling:
`txn.appendAndSync` held the cap-1 writer semaphore from `Begin` to
`Commit` and called one `wal.Sync()` per transaction, hard-capping
throughput at ≈ 1/fsync-latency commits per second regardless of core
count (a real fsync is ≈ 3.5 ms against an encode-only ≈ 312 ns).

The new `wal.Writer.SyncGroup()` implements a PostgreSQL-`XLogFlush`-style
leader/follower coalesced fsync. Each committer records its durability
watermark; one **leader** flushes and fsyncs the whole buffered suffix
once (releasing the WAL mutex across the slow fsync so followers keep
appending), publishes the durable watermark, and wakes the followers;
each **follower** whose watermark is covered returns success with **no
I/O of its own**. `Commit` now appends its op frames and the `OpCommit`
marker under the semaphore (on-disk frame order unchanged), releases the
semaphore, runs `SyncGroup` with the semaphore free (the coalescing
window), then applies the in-memory change strictly in WAL sequence
order.

**Measured (`BenchmarkCommitConcurrent`, count = 8):**

| Goroutines | Before | After | Delta |
|-----------:|-------:|------:|-------|
| 1 | 3.683 ms | 3.683 ms | flat (p = 0.959) — **no regression** |
| 8 | 3.67 ms | 950 µs | **−74.15 %** |
| 64 | 3.68 ms | 120 µs | **−96.72 %** |
| 256 | 3.67 ms | 31 µs | **−99.16 % (≈ 118× throughput)** |

The **storage-engine auditor certified the merge**: Atomicity (the WAL
stream is byte-identical and recovery is unchanged), Consistency, Isolation
(the in-memory apply runs only after durability and only in sequence
order — durable-before-visible preserved), and Durability (the commit acks
only after the covering fsync; a failed group fsync fails **every** member
of the group) are all preserved.

### Parallel pull-formulation PageRank over reverse-CSR (`search/centrality/`, #1513)

PageRank's per-iteration SpMV was a single-goroutine push scatter. It now
runs the **pull** formulation —
`next[v] = baseShare + d·Σ_{u∈in(v)} cur[u]/outdeg[u]` — over a
structure-only reverse-CSR, parallelised across a **persistent** worker
pool (created once before the power-iteration loop and reused every
iteration), partitioned by approximately equal in-edge count for
power-law load balancing.

Each `next[v]` is computed independently with no write contention, and
every vertex sums its in-edges in the fixed reverse-CSR order — the same
increasing-source-id order the serial push path accumulates in — so the
result is **bit-for-bit identical to the serial path regardless of
`GOMAXPROCS` or scheduling** (proven cross-process: `GOMAXPROCS=1` vs `8`
produce byte-identical hex `float64` ranks for all 8 192 vertices and an
identical iteration count). The path is gated behind
`live >= 2048 && GOMAXPROCS > 1`; smaller graphs and single-core runs take
the unchanged serial push path and pay neither the transpose nor any
goroutine overhead.

**Measured (BarabasiAlbert directed, 30 iterations):** 50K-node / 800K-edge
**1.77×** (13.88 ms → 7.86 ms); 100K-node / 3.2M-edge **1.68×** (37.5 ms →
22.3 ms); with the transpose amortised over 200 iterations the SpMV kernel
alone reaches **2.40×** (90.7 ms → 37.9 ms) — the honest single-socket
ceiling for this latency-bound random-gather SpMV. `GOMAXPROCS=1` shows
**no regression** (the serial path is byte-identical). The go-developer
specialist certified the concurrency design as race-free, leak-free, and
shutdown-safe.

### Range-predicate B+tree index seek (`cypher/`, #1505)

A range predicate (`n.p > x`, `>=`, `<`, `<=`, or a two-sided AND) on a
property backed by a **bound string btree index** now builds a
`NodeByIndexRangeScan` in place of the `NodeByLabelScan` child, with the
original `Selection` filter retained on top (seek-superset plus residual
refilter). A prerequisite fix makes a Cypher
`CREATE INDEX … {indexType:'btree'}` a **bound, backfilled,
self-maintaining** B+ tree index — it was previously registered empty
with a no-op `Apply` and never populated.

The seek is conservative: it fires only for a string index and string
bound, only when the in-range count is ≤ 10 % selective and the labelled
set has ≥ 1 024 nodes, and otherwise falls back to scan-plus-filter.
**Measured (50K `:RSPerson`, 0.5 %-selective range):** **−99.11 % time
(≈ 114×), −98.95 % B/op (≈ 95×)**, with **byte-identical** allocations on
all curated entries (the seek never triggers on the LDBC IC queries). A
differential test proves an identical result multiset seek-on vs seek-off
across open/closed/two-sided/empty/no-match ranges, mixed-typed
properties, `ORDER BY`, `count()`, and post-mutation self-maintenance.

### Hash join for disconnected equi-join patterns (`cypher/exec/`, #1506)

A disconnected multi-pattern `MATCH` joined by an equality predicate
(`MATCH (a:A),(b:B) WHERE a.x = b.y …`) previously degraded to a
nested-loop Cartesian product plus an equi-join filter (O(|A|·|B|)). It is
now built as a new `exec.HashJoin` (build a hash table on the inner arm
keyed by the join column, probe with the outer; O(|A|+|B|)), under a
structural trigger, a build-arm size floor, and an order-safety guard.
**Measured:** ≈ **93× faster, ≈ 95× less memory** on the targeted shape
(1.75 ms / 3.28 MB vs 162 ms / 311 MB). A differential test proves an
identical result multiset hash-join-on vs hash-join-off across multi-row,
no-match, NULL keys, cross-type int/float keys, NaN keys, residual
predicates, and `id()`-equi-joins; result identity rests on excluding
NULL/NaN keys (3VL), a canonical key hash that folds integral floats to
the integer hash, and per-bucket re-verification with `expr.Value.Equal`.

### Cypher read-path allocation cuts (`cypher/`, `graph/lpg/`, #1499–#1503)

- **Column-oriented (SoA) result rows** (#1499) — `materialize` copies
  each row's positional values into one flat backing slice instead of a
  `map[string]Value` per row; the public per-row map is built lazily into
  a single reused scratch only when `Record()` is called. **IC1 −32.4 %
  time / −60.9 % B/op / −25.6 % allocs.**
- **Deferred node materialisation for scalar-only projections** (#1500) —
  a node read only through scalar accesses no longer eagerly copies its
  full property map and label set per row. Isolated scalar projection
  **−42 % time / −39 % allocs.**
- **Single-map `WITH`/`RETURN` projection** (#1501) — the redundant
  per-row map clone in the general projection path is removed. Isolated
  **−20.1 % time / −16.9 % allocs.**
- **Shared node property/label views across the lpg→expr→wire seam**
  (#1502) — a returned node's property map is copied out of storage
  exactly once. Whole-node return path **−20.2 % B/op.**
- **Lock-free copy-on-write metadata name registry** (#1503) — label and
  property-key resolution no longer takes an RWMutex per resolved id.
  **−81.57 % time at 8-way contention** on the isolated metadata-read
  benchmark; the curated set is byte-identical on allocations and B/op.

### Durability and bulk-load hardening (`store/`, #1508–#1512)

A pre-sized partitioned-parallel bulk loader (#1512); a non-blocking
LSN-watermarked checkpoint with WAL prefix truncation (#1508);
elimination of the per-WAL-frame double allocation and copy (#1509); and
`fdatasync` for the per-commit WAL sync on Linux (#1510). A CRC32C
strategy microbenchmark confirmed the incremental `Update` path already
uses the hardware path (#1511).

## Security-relevant hardening

This release ships **no toolchain change** — the Go toolchain remains
**go1.26.4**, and `govulncheck ./...` is clean. It closes two further
security audits, both fully remediated and both shipping a strict
regression battery.

### SEC-2026-06-14b (9 findings, #1480–#1489)

The dominant theme was that the prior audit's "bound the eager allocation
before the integrity gate" discipline had **not been generalised** to
sibling decoders. The full report is in
[docs/security-audit-2026-06-14b.md](../docs/security-audit-2026-06-14b.md).
All reachable from untrusted input:

- **Untrusted-artefact allocation and recursion bounds** — the btree
  index decoder (#1480) and label index decoder (#1481) now clamp eager
  allocation to `min(count, 1<<20)` and grow on demand; the snapshot
  component readers thread `FileEntry.Size` through an `io.LimitReader`
  so the `append` loop cannot grow past the on-disk size (#1486); and the
  property-value decoder rejects a nested `PropList`, closing an
  unbounded-recursion stack-exhaustion path (#1488).
- **Cypher string-byte budget** — string concatenation is charged against
  a per-evaluation **byte** budget before allocation (#1482).
- **Bolt `tx_timeout` overflow** — converted overflow-safely; a
  non-positive/overflowing value falls back to the server default so the
  wall-clock writer-lock reaper is always armed (#1484).
- **Cartesian-product notification and cancellation** — a plan-time
  notification plus a locked-in deadline-cancellation guarantee (#1483).
- **Bolt NOOP keep-alive** discarded silently (#1485) and **DOT
  reserved-keyword** ids quoted (#1489).

### SEC-2026-06-14c (7 findings, #1490–#1496)

**No Critical finding and no prior-fix regression**; the residuals were
arithmetic edge-cases in Cypher built-ins and defence-in-depth gaps. The
full report is in
[docs/security-audit-2026-06-14c.md](../docs/security-audit-2026-06-14c.md):

- **`substring()` integer overflow** — the end bound is computed
  overflow-safely, returning the conforming truncated tail with no panic
  (#1492).
- **`percentileCont()` NaN guard** — a non-finite percentile parameter is
  rejected, and the continuous aggregator clamps its index defensively
  (#1493).
- **`replace()` amplification bound** — `replace(s, '', r)` computes its
  worst-case output size overflow-safely and returns a typed
  `NumberOutOfRange` budget error before allocating (#1494).
- **txn PropList decoder clamp** brings the third parallel decoder into
  lockstep with its siblings (#1490); the **Bolt reader goroutine** gains
  a panic-recovery boundary (#1491).
- **Supply chain** — a stale `.goreleaser.yaml` Dependabot comment was
  corrected with a regression gate (#1495), and `gen_tck_tzdata.sh` now
  pins the SHA-256 of the IANA tzdata tarball (#1496).

Both compliance invariants held throughout (openCypher TCK 3 897 / 3 897,
ACID), and `go test -race`, golangci-lint, staticcheck and `govulncheck`
are clean.

## Upgrade notes

This release is **API-additive** over `v0.3.0`: no exported identifier
was added, removed, or renamed, there is **no breaking API change**, and
there is **no behavioural change** to any documented API contract.
Upgrading is a drop-in `go get`:

```bash
go get github.com/FlavioCFOliveira/GoGraph@v0.3.1
```

Two points are worth noting, neither of which requires action:

1. **Cartesian-product notifications are now available** (opt-in). A
   disconnected multi-pattern `MATCH` surfaces a Cartesian-product
   notification through the new `Result.Notifications()` accessor and the
   Bolt PULL `SUCCESS` metadata. This is purely informational; existing
   callers that ignore the metadata are unaffected (#1483).
2. **A Cypher `btree` index now actually backs range queries.** A
   `CREATE INDEX … {indexType:'btree'}` index is now bound, backfilled,
   and self-maintaining (it was previously registered empty), and range
   predicates over a selective string btree index are served by a fast
   index seek. This is a performance improvement with identical results;
   no query needs to change.

Because the public API is pre-1.0, continue to **pin the exact version**
you depend on.
