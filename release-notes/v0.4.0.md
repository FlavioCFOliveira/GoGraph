## Overview

GoGraph v0.4.0 is the sixth published release of GoGraph, a Go module
for graph **persistence**, **manipulation**, and — above all — **fast
search**, designed to scale from in-memory graphs to graphs that exceed
RAM.

This is a pre-1.0 **MINOR** release. Its headline is a **new public
API** — `(*cypher.Engine).BeginReadTx` — that opens a **lock-free,
read-only explicit transaction** and lifts concurrent read throughput by
roughly **2×**. The release also lands targeted **Cypher read-path
performance work** and the **first publication of the
deterministic-simulation-testing (DST) harness**, the test/CI
infrastructure that found and fixed two latent defects shipped in this
release.

The bump is **MINOR** under [Semantic Versioning](https://semver.org/):
`BeginReadTx` and its companion sentinel `ErrWriteInReadOnlyTx` are
**purely additive** over `v0.3.2` — no exported identifier was removed or
renamed, and there is **no breaking change**. As a `0.y.z` release the
public API remains unstable: it may change without a major-version bump
while the module matures toward `1.0.0`. Pin the exact version you depend
on.

Both compliance guarantees that define the project continue to hold
without regression at this version:

- **100 % openCypher TCK-compliant at the execution level** — every
  scenario in `cypher/tck/features/` passes, with no `failed`,
  `undefined`, or `pending` step. The execution gate stands at
  **3 897 / 3 897 scenarios** and **16 006 / 16 006 steps**
  (error-fidelity gate 121). The regression gate
  (`tckExecutionBaseline = 3897` in `cypher/tck/runner_test.go`) rejects
  any change that lowers the count.
- **100 % ACID-compliant** — Atomicity, Consistency, Isolation, and
  Durability hold across the in-memory engine and every persistence
  backend; durability is verified by the WAL recovery tests in
  `store/wal` and `store/recovery` and by the deterministic
  crash-injection battery in `internal/crashinject`.

Install with:

```bash
go get github.com/FlavioCFOliveira/GoGraph@v0.4.0
```

The Go module path is `github.com/FlavioCFOliveira/GoGraph` with **no**
`/vN` suffix, which is Semantic-Import-Versioning-correct for a `0.x`
line. The Go toolchain remains **go1.26.4** (unchanged from `v0.3.2`),
and `govulncheck ./...` stays clean.

For the full, per-change log see [CHANGELOG.md](../CHANGELOG.md).

## Highlights

- **New API: lock-free read-only explicit transactions** —
  `(*cypher.Engine).BeginReadTx(ctx)` opens a read-only transaction that
  acquires neither the writer serialisation, the visibility barrier, nor
  a WAL transaction, so reads never block behind a concurrent writer.
  Concurrent read throughput roughly **doubles** and scales with core
  count.
- **Cypher read-path performance** — relationship reconstruction is up to
  **−88 % sec/op / −96 % B/op** on dense graphs (per-query forward-CSR
  cache, #1574), and scalar projection is **−42 % B/op / −27 % allocs/op**
  (non-escaping `RowContext` pooling, #1575).
- **Deterministic-simulation-testing (DST) harness, first publication** —
  a TigerBeetle-VOPR-modelled simulator (`internal/sim`, `cmd/sim`) for
  reproducible, seed-driven fault and concurrency testing; it found and
  fixed two latent defects now shipped in this release.

## What changed

### New public API — `BeginReadTx` (#1573)

`(*cypher.Engine).BeginReadTx(ctx context.Context) (*ExplicitTx, error)`
opens a **read-only explicit transaction**. Unlike `BeginTx`, the
returned handle acquires **none** of the engine writer serialisation, the
visibility barrier, or a WAL transaction — a read-only transaction has no
durability obligation and never serialises behind, or blocks, a
concurrent writer.

Semantics:

- **Writes are rejected before execution.** Any statement run through the
  handle that contains a writing clause, or is DDL, is rejected with the
  new exported sentinel `cypher.ErrWriteInReadOnlyTx` **before** it
  executes. This guard is precisely what makes the lock-free path safe: a
  write would otherwise run with no writer lock, no barrier, and no WAL.
- **Read-committed isolation across statements.** Each permitted read
  routes through the engine's concurrent read path and takes its own
  per-statement snapshot, so reads observe the latest committed state
  across the statements of the transaction — matching Neo4j's default.
- **Teardown-only finish.** The caller must still finish the handle with
  exactly one `Commit` or `Rollback`; on a read-only handle both are
  no-ops, because nothing was acquired.
- **Bolt integration.** Over the Bolt protocol, `BEGIN` with `mode="r"`
  now routes through `BeginReadTx`. `ErrWriteInReadOnlyTx` maps to
  `Neo.ClientError.Request.Invalid`, and the message is forwarded to the
  client as a client-fault.

**Motivation.** Block-profiling of a Bolt read workload attributed
**83 %** of all blocking time to the store single-writer semaphore that
`BeginTx` held for the entire duration of an explicit transaction — even
when that transaction only read. `BeginReadTx` removes that contention
for the read-only case.

Minimal usage:

```go
tx, err := engine.BeginReadTx(ctx)
if err != nil {
    return err
}
defer tx.Rollback() // teardown-only no-op for a read-only handle

res, err := tx.Exec(ctx, "MATCH (n:Person) RETURN n.name", nil)
if err != nil {
    return err
}
_ = res

// A writing statement is rejected before execution:
if _, err := tx.Exec(ctx, "CREATE (:Person)", nil); errors.Is(err, cypher.ErrWriteInReadOnlyTx) {
    // expected: writes are not permitted on a read-only transaction
}
```

### Cypher read-path performance

- **Per-query forward-CSR cache in relationship reconstruction (#1574).**
  `edgeHandleAtFwdPos` / `edgeInstanceIdxFor` previously rebuilt the
  entire forward CSR (`O(V+E)`) on **every** result row to read one slot,
  making reconstruction `O(R·(V+E))` per query. The forward CSR snapshot
  is now built once per query and reused, with all slot reads served from
  the cached snapshot under the query's single read lock. On
  `MATCH (a)-[r]->(b) RETURN count(r)` over a multigraph: **dense graph
  −88.2 % sec/op, −95.9 % B/op**; small graph −73.6 % sec/op, −83.3 %
  B/op. Certified by `graph-theory-expert`, `storage-engine-auditor`, and
  `go-developer`.
- **Non-escaping `RowContext` pooling (#1575).** The per-row context map
  is recycled through a `sync.Pool` at the two non-escaping evaluation
  sites (the `Filter` predicate closure and the scalar-projection
  evaluator). On `MATCH (n:N) WHERE n.i>=0 AND n.j>=n.i RETURN n.i,n.j`:
  **−42.1 % B/op, −26.6 % allocs/op, −11.5 % sec/op**. The pooling is
  gated on the existing analysis flag that bails on any expression kind
  that could retain the context past the call, so a recycled map is never
  referenced after the row is evaluated; values that escape into a result
  row are independently owned and never reused. Certified by
  `go-developer`; `rust-perf-engineer` consulted.
- **Brandes betweenness zero-allocation predecessor arena (#1515)** and a
  **configurable DST invariant-check cadence (#1576)** round out the
  performance work.

See [docs/benchmarks/v0.4.0.md](../docs/benchmarks/v0.4.0.md) for the
full figures.

### Deterministic-simulation-testing (DST) harness (test/CI infrastructure)

This release publishes the DST harness (`internal/sim`, `cmd/sim`) for
the first time — a TigerBeetle-VOPR-modelled deterministic simulator:

- seed-driven virtual clock and an in-memory faulting disk with real WAL
  recovery;
- a graph oracle and invariant checkers (including full
  index-vs-base-data consistency);
- crash-and-recovery cycles and a real in-process Bolt wire path;
- a named scenario catalogue with deterministic trace recording, scripted
  replay, and `ddmin` shrinking to a minimal failing reproducer;
- a reproducible bounded-worker swarm; and
- upgrade, cross-release, and differential harnesses, plus a metrics
  oracle.

The `cmd/sim` CLI exposes `-swarm`, `-scenario`, `-replay`, and
`-check-every`. This is **test and CI infrastructure**, not a public
module API — it is documented here because it is the provenance of the
two fixes below and a new layer of the project's reliability evidence.

### Fixes found via the simulator

- **Bolt illegal-transition error names the originating state** (commit
  `82d98af`). An illegal protocol transition into `FAILURE` now reports
  the state the connection was actually in, rather than always reporting
  `FAILED`.
- **`DROP CONSTRAINT` by name is atomic and never a silent no-op**
  (#1556). Dropping a constraint by name now drops the constraint **and**
  its backing index atomically; a previously fail-silent path that left
  the constraint in place is fixed. Schema DDL is not covered by the
  openCypher TCK, so the execution-conformance count is unaffected.

## Upgrade notes

This release is **API-additive** over `v0.3.2`: it adds
`(*cypher.Engine).BeginReadTx` and `cypher.ErrWriteInReadOnlyTx`, and
removes or renames nothing. There is **no breaking change** and **no
required migration**. Upgrading is a drop-in `go get`:

```bash
go get github.com/FlavioCFOliveira/GoGraph@v0.4.0
```

**Adopting `BeginReadTx`** is optional. If you currently open read-only
work through `BeginTx`, you can switch read-only transactions to
`BeginReadTx` to remove writer-serialisation contention and gain the
concurrent-read throughput improvement. Note the two behavioural
differences relative to `BeginTx`:

- a writing or DDL statement is rejected with `ErrWriteInReadOnlyTx`
  rather than executed; and
- isolation is **read-committed across statements** (each `Exec` sees the
  latest committed state) rather than a single snapshot for the whole
  transaction.

Because the public API is pre-1.0, continue to **pin the exact version**
you depend on.
